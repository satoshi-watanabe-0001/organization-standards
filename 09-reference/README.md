# リファレンス資料 (Reference Materials)

## 📋 概要

このディレクトリには、開発作業中に素早く参照できるリファレンス資料が含まれています。用語集、クイックリファレンス、エスカレーションフロー、意思決定マトリックス、チェックリスト集など、日常的な開発業務を効率化するための資料を提供します。

**重要度**: Tier 4（効率化・運用）  
**対象**: 全開発者、プロジェクトマネージャー、Devin AI  
**更新頻度**: 四半期ごとのレビュー推奨

### 🚨 3. escalation-matrix.md

**概要**: 問題発生時のエスカレーションフローと連絡先マトリックス

**主要内容**:
- **エスカレーションレベル**:
  - **L1（初期対応）**: チーム内解決、技術リード相談
  - **L2（専門家対応）**: 専門チーム介入、アーキテクト相談
  - **L3（経営判断）**: 経営層エスカレーション、外部専門家起用
- **問題カテゴリ別フロー**:
  - **技術的問題**: バグ、パフォーマンス、アーキテクチャ
  - **セキュリティインシデント**: 脆弱性発見、侵入検知
  - **コンプライアンス違反**: 規制違反、監査指摘
  - **プロセス違反**: 標準逸脱、承認漏れ
  - **リソース問題**: 予算超過、人員不足、期限遅延
- **連絡先情報**:
  - 技術リード、アーキテクト
  - セキュリティチーム、コンプライアンス担当
  - プロジェクトマネージャー、経営層
  - オンコール体制、緊急連絡先
- **エスカレーション基準**:
  - 重要度判定マトリックス（Critical/High/Medium/Low）
  - 影響範囲評価（ユーザー数、売上影響）
  - タイムライン（即時/24時間以内/1週間以内）
- **エスカレーション手順**:
  1. 問題の初期評価
  2. 重要度・影響範囲の判定
  3. 適切なエスカレーションレベルの選択
  4. 関係者への連絡
  5. 記録と追跡

**使用方法**:
- 問題発生時の即座の判断基準
- 適切な担当者・チームへの迅速な連絡
- エスカレーション記録の標準化

**対象者**: 全開発者、プロジェクトマネージャー、Devin AI  
**外部文書URL**: [作成済み外部文書]

### 🎯 4. decision-matrix.md

**概要**: 技術選定、アーキテクチャ決定時の意思決定マトリックス集

**主要マトリックス**:
- **技術選定マトリックス**:
  - **フロントエンドフレームワーク選定**:
    - 評価軸: 学習曲線、パフォーマンス、エコシステム、コミュニティサポート
    - 候補: React、Vue、Angular
    - スコアリング手法、重み付け
  - **バックエンド言語/フレームワーク選定**:
    - 評価軸: 開発速度、パフォーマンス、スケーラビリティ、人材確保
    - 候補: Node.js/Express、Python/FastAPI、Java/Spring Boot
  - **データベース選定**:
    - 評価軸: データモデル適合性、スケーラビリティ、運用コスト
    - 候補: PostgreSQL、MySQL、MongoDB、Redis
- **アーキテクチャ決定マトリックス**:
  - **モノリス vs マイクロサービス**:
    - 評価軸: 複雑度、チーム規模、デプロイ頻度、スケーラビリティ要件
    - 意思決定フローチャート
  - **同期 vs 非同期通信**:
    - 評価軸: レイテンシ要件、一貫性要件、複雑度許容
    - REST vs gRPC vs Message Queue
  - **SQL vs NoSQL**:
    - 評価軸: データ構造、トランザクション要件、スケール要件
- **クラウドプロバイダー選定**:
  - 評価軸: コスト、サービス充実度、既存スキル、ベンダーロックイン
  - 候補: AWS、GCP、Azure
- **セキュリティアプローチ選定**:
  - 認証方式: Session vs JWT vs OAuth 2.0
  - 暗号化方式: 対称鍵 vs 非対称鍵
- **デプロイメント戦略選定**:
  - Blue-Green、Canary、Rolling Update
  - 評価軸: ダウンタイム許容度、ロールバック容易性

**使用方法**:
1. 意思決定が必要な領域を特定
2. 該当するマトリックスを参照
3. 評価軸に基づいてスコアリング
4. 重み付けを適用して総合評価
5. 決定内容と根拠を文書化

**対象者**: アーキテクト、技術リード、プロジェクトマネージャー、Devin AI  
**外部文書URL**: [作成済み外部文書]

**用語確認**:
```
「glossary.mdで'CQRS'の定義を確認してください」
```

**コマンド参照**:
```
「quick-reference.mdからKubernetesのログ確認コマンドを取得して実行してください」
```

**エスカレーション判断**:
```
「現在のセキュリティ問題をescalation-matrix.mdに基づいて評価し、
適切なエスカレーションレベルを判定してください」
```

**技術選定支援**:
```
「decision-matrix.mdのデータベース選定マトリックスを使用して、
現在のプロジェクト要件に最適なデータベースを推奨してください」
```

```
「checklists.mdのリリース前チェックリストに基づいて、
現在のプロジェクトの準備状況を評価してください」
```

### 人間開発者による使用

**日常的な参照**:
- 不明な用語に遭遇したら即座にglossary.mdを確認

**意思決定支援**:
- 技術選定時はdecision-matrix.mdのマトリックスを活用
- 客観的評価基準でチーム合意形成
- 決定内容と根拠を文書化

**品質保証**:
- 各フェーズでchecklists.mdの該当チェックリストを使用
- チェックリストの完了を品質ゲートとして設定
- 未完了項目を追跡

## 🔍 検索・ナビゲーション方法

### 用語検索（glossary.md）
1. アルファベット順索引から該当セクションへ移動
2. ブラウザの検索機能（Ctrl+F / Cmd+F）で用語検索
3. 関連用語へのクロスリファレンスをたどる

### コマンド検索（quick-reference.md）
1. カテゴリ（Git、Docker、Kubernetes等）から絞り込み
2. ユースケース（ログ確認、デバッグ等）から検索
3. コピー&ペースト可能な形式で記載

### エスカレーション判断（escalation-matrix.md）
1. 問題カテゴリ（技術、セキュリティ、コンプライアンス等）を特定
2. 重要度判定マトリックスで評価
3. 該当するエスカレーションレベルのフローを参照
4. 連絡先情報で適切な担当者を確認

### 意思決定（decision-matrix.md）
1. 決定が必要な領域（技術選定、アーキテクチャ等）を特定
2. 該当するマトリックスを参照
3. 評価軸に基づいてスコアリング
4. 総合評価で最適解を導出

### チェックリスト活用（checklists.md）
1. 現在のフェーズ（開始、開発、リリース等）を特定
2. 該当するチェックリストを取得
3. 項目を順次確認しながら作業
4. 完了状況を記録
