# PBI受領時チェックリスト

> Product Backlog Item（PBI）を受け取った際の初期確認項目

**目的**: PBIが実装可能な状態かを確認し、Phase 0（要件分析）を開始する前の準備を整える  
**使用タイミング**: PBIがアサインされた直後、Phase 0開始前  
**参照元**: [AI-PRE-WORK-CHECKLIST.md](../../00-guides/AI-PRE-WORK-CHECKLIST.md)

---

## ✅ 使い方

このチェックリストは、新しいPBIを受け取った際に、最初に実施する確認項目です。

### チェック方法
1. PBIのJIRAチケットを開く
2. 以下の各項目を順番に確認
3. すべて完了したら、Phase 0（要件分析）へ進行

### 不明点がある場合
- PBIの作成者に確認
- プロダクトオーナーに質問
- 必要な情報が揃うまで待機

---

## 📋 PBI基本情報確認

### 必須情報の存在確認

- [ ] PBIタイトルが明確である
- [ ] PBI説明文が記載されている
- [ ] 受入基準（Acceptance Criteria）が定義されている
- [ ] 優先度（Priority）が設定されている
- [ ] PBIタイプが識別できる（新規機能/バグ修正/リファクタリング等）
- [ ] 関連するエピック・ストーリーが紐付いている（該当する場合）
- [ ] 添付ファイル・参考資料が確認できる（該当する場合）

### PBIタイプの判別

以下のいずれかに該当することを確認:

- [ ] **新規プロジェクト**: ゼロから新規プロジェクトを立ち上げ
- [ ] **機能追加**: 既存システムへの新機能追加
- [ ] **バグ修正**: 既存機能の不具合修正
- [ ] **リファクタリング**: コード品質改善（機能変更なし）
- [ ] **設計変更**: アーキテクチャ・設計の大幅変更
- [ ] **Hotfix**: 本番障害の緊急修正
- [ ] **実験的機能**: PoC・試験的実装

---

## 🎯 要件明確性の確認

### ビジネス要件

- [ ] 「何のために」この機能が必要かが説明されている
- [ ] ビジネス上の価値が理解できる
- [ ] 対象ユーザー・ステークホルダーが明確である
- [ ] 期待される成果が定義されている

### 機能要件

- [ ] 「何を実装するか」が具体的に記載されている
- [ ] 入力・出力が明確である
- [ ] 画面・UI要件が定義されている（該当する場合）
- [ ] データ要件が明確である（該当する場合）

### 非機能要件

- [ ] パフォーマンス要件が定義されている（該当する場合）
- [ ] セキュリティ要件が明確である（該当する場合）
- [ ] 可用性要件が記載されている（該当する場合）
- [ ] スケーラビリティ要件が定義されている（該当する場合）

---

## 📅 スケジュールと制約

### タイムライン確認

- [ ] 期限（Due Date）が設定されている
- [ ] 期限が現実的である（見積もりと比較）
- [ ] 依存関係のあるPBIがブロックしていない
- [ ] 必要なリソースが確保できる見込みがある

### 制約条件の確認

- [ ] 技術的制約が明確である（使用技術、ライブラリ制限等）
- [ ] ビジネス制約が理解できる（予算、コンプライアンス等）
- [ ] 環境制約が確認できる（インフラ、ツール等）

---

## 🔗 関連情報の確認

### 参照ドキュメント

- [ ] 関連する設計書・仕様書が存在する
- [ ] 参照ドキュメントにアクセスできる
- [ ] ドキュメントの最新版が確認できる

### 関連PBI・課題

- [ ] 関連するPBIがリンクされている
- [ ] ブロッカーとなるIssueがないか確認
- [ ] 過去の類似PBIを検索・確認

---

## ✋ Go/No-Go判断

### Phase 0開始可否の判断

すべての必須項目が確認できた場合:
- [ ] **Go**: Phase 0（要件分析）を開始できる

以下のいずれかに該当する場合は、No-Go（開始保留）:
- [ ] PBIの必須情報が不足している → 作成者に確認依頼
- [ ] 要件が不明確すぎる → プロダクトオーナーと要件整理
- [ ] 依存関係がブロックしている → ブロッカー解消まで待機
- [ ] リソースが確保できない → リソース調整
