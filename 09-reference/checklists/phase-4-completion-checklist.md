# Phase 4 完了チェックリスト

> コードレビューと品質保証完了確認

**目的**: Phase 4のレビューとQAが完了し、本番デプロイの準備が整っていることを確認  
**使用タイミング**: Phase 2B（事後実装設計）またはPhase 5（デプロイ）への移行前  
**参照元**: [phase-4-review-qa-guide.md](../../00-guides/phase-guides/phase-4-review-qa-guide.md)

---

## ✅ 使い方

このチェックリストは、Phase 4（レビュー・QA）のすべてのタスクが完了し、次のフェーズに移行する前に使用します。

### チェック方法
1. 各項目を順番に確認
2. 完了している項目には `[x]` をマーク
3. すべての項目が完了したら、次のフェーズへ進行可能

### 未完了項目がある場合
- 該当項目の担当者に確認
- 必要に応じてタスクを追加・完了
- すべて完了するまで次フェーズへ進まない

## 🎯 Phase 4 の成果物

### 必須成果物

- [ ] **コードレビューレポート**
- [ ] **テスト実行結果レポート**
- [ ] **品質メトリクスレポート**
### 推奨成果物

- [ ] **パフォーマンステストレポート**（該当する場合）
- [ ] **セキュリティスキャンレポート**
- [ ] **不具合管理票**（発見された場合）

## 🔄 Phase 4 のワークフロー: 7ステップ

### **Step 4.1: コードレビュー** (30-60分)

- [ ] コーディング標準に準拠しているか
- [ ] 設計原則（SOLID等）に従っているか
- [ ] セキュリティ脆弱性がないか
- [ ] パフォーマンス上の問題がないか
- [ ] エラーハンドリングが適切か
- [ ] ログ出力が適切か
- [ ] コメント・ドキュメントが十分か
- [ ] テストが適切に実装されているか
- [ ] 自動チェック（リンター、フォーマッター）の実行
- [ ] 手動レビュー（ロジック、設計、セキュリティ）
- [ ] フィードバックの記録と対応
### **Step 4.1.5: CI設定とCI実行結果の検証** ⭐NEW (10-15分)

- [ ] PRのCI実行状態確認
- [ ] CI品質ゲートの実行確認
- [ ] カバレッジ閾値達成確認
### **Step 4.1.6: PR説明文・言語チェック** ⭐NEW (5分)

- [ ] PRタイトルの言語確認
- [ ] CI品質ゲート（言語チェック）の通過確認
- [ ] PRタイトルが日本語で記載されている
- [ ] PR説明文が日本語で記載されている
- [ ] 変更内容の概要が日本語で明確に記載されている
- [ ] テスト結果が日本語で記載されている
- [ ] レビュアーへの補足が日本語で記載されている
- [ ] CI品質ゲート「PR Language Check」が✅合格している
- [ ] **PRを修正する（推奨）**
- [ ] **PRを再作成する**

## Test Results

### **Step 4.1.7: PR記載レベル品質チェック** ⭐NEW (5分)

- [ ] PR説明文の記載レベル確認
- [ ] 必須セクションの存在確認
- [ ] チェックリスト確認率の検証
- [ ] CI品質ゲート（記載レベルチェック）の通過確認
- [ ] 必須セクション7つがすべて記載されている
- [ ] 変更内容の概要が50文字以上（推奨100文字以上）
- [ ] チェックボックス確認率が70%以上（推奨85%以上）
- [ ] 日本語で記載されている
- [ ] 関連チケット番号が明確（#123, PBI-456等）
- [ ] CI品質ゲート「PR Description Quality Gate」が✅合格している
- [ ] **PRを修正する（推奨）**
- [ ] **具体的な改善方法**

## テスト

### **Step 4.2: テスト戦略の確認** (10-15分)

- [ ] 必要なテストレベルの決定
- [ ] どのテストレベルが必要か確認（ユニット・統合・E2E）
- [ ] カバレッジ目標を確認（通常80%以上）
- [ ] テストデータの準備方法を確認
- [ ] テスト環境の準備状況を確認
### **Step 4.3: ユニットテスト** (60-90分)

- [ ] ユニットテストケースの作成
- [ ] テスト実行とカバレッジ測定
- [ ] すべての公開メソッド/関数にテストがあるか
- [ ] 正常系と異常系の両方をテストしているか
- [ ] エッジケースをテストしているか
- [ ] モック/スタブを適切に使用しているか
- [ ] テストカバレッジが目標値に達しているか
- [ ] すべてのすべてのテストが合格している
### **Step 4.4: 統合テスト** (45-60分)

- [ ] 統合テストシナリオの作成
- [ ] コンポーネント間の連携をテストしているか
- [ ] データベース操作をテストしているか
- [ ] 外部APIとの連携をテストしているか
- [ ] トランザクション処理をテストしているか
- [ ] すべてのすべてのテストが合格している
### **Step 4.5: E2Eテスト** (60-90分、該当する場合)

- [ ] E2Eテストシナリオの作成
- [ ] 主要なユーザーフローをテストしているか
- [ ] UIの表示と操作をテストしているか
- [ ] エンドツーエンドのデータフローをテストしているか
- [ ] エラーシナリオをテストしているか
- [ ] すべてのすべてのテストが合格している
### **Step 4.6: コード品質チェック** (15-20分)

- [ ] リンター/フォーマッターのチェック
- [ ] リンターエラーがゼロであるか
- [ ] フォーマッターチェックがパスしているか
- [ ] 静的解析ツールの警告を確認
- [ ] 循環的複雑度が許容範囲内か
- [ ] コードの重複がないか
- [ ] 技術的負債スコアを確認
### **Step 4.7: パフォーマンス・セキュリティテスト** (30-60分、オプション)

- [ ] パフォーマンステストの実施
- [ ] セキュリティスキャンの実施
- [ ] レスポンスタイムが目標値以内か
- [ ] スループットが目標値を達成しているか
- [ ] メモリリークがないか
- [ ] セキュリティ脆弱性スキャンを実施
- [ ] 依存関係の脆弱性をチェック

## ドキュメントコメント不足（却下）

### 4.8.6 レビュー手順

- [ ] 自動チェック実行（5分）
- [ ] 手動レビュー（30-60分）
- [ ] 承認/却下/改善要求の判断
### 4.8.8 却下基準のまとめ

- [ ] 言語別Linterでエラー・警告が1件でもある
- [ ] ドキュメント生成ツールが失敗する
- [ ] パブリックAPIのカバレッジが100%未満
- [ ] ファイルヘッダーコメントが不足
- [ ] パブリッククラスにコメントが不足
- [ ] パブリック関数にコメントが不足
- [ ] パラメータ・戻り値・例外の説明が不足
- [ ] "Why"の説明が不足（改善要求）
- [ ] ビジネスロジックの説明が不足（改善要求）
- [ ] 複雑なロジックへのコメントが不足（改善要求）

## 📚 不具合管理（継続的）

### 不具合が見つかった場合

- [ ] **不具合の記録**: JIRA等に記録
- [ ] **優先度付け**: Critical / High / Medium / Low
- [ ] **修正**: Phase 3に戻って修正実施
- [ ] **再テスト**: 修正箇所のテスト実施
- [ ] **回帰テスト**: 影響範囲の確認

## 🔍 実践的な活用方法

### 基本的なワークフロー

- [ ] **コードレビューから開始**
- [ ] **テスト戦略を確認してテスト実施**
- [ ] **品質チェックを実施**
- [ ] **必要に応じて追加テスト**
