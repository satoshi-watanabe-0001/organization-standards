## 作成タイミングマトリクス

### 設計フェーズのタイムライン

```
Week 1-2: プロジェクト開始・技術提案
├─ 【成果物】技術提案書 (Technical Proposal)
│  ├─ 作成者: テックリード、アーキテクト
│  ├─ レビュー: エンジニアリングチーム、PM
│  └─ 承認: CTO、エンジニアリングマネージャー
│
└─ 【成果物】初期ADR (重要な技術選定がある場合)
   └─ 作成者: アーキテクト

Week 3-4: 詳細設計
├─ 【成果物】設計書 (Design Document)
│  ├─ 作成者: リードエンジニア、担当エンジニア
│  ├─ レビュー: テックリード、アーキテクト
│  └─ 承認: エンジニアリングマネージャー
│
├─ 【成果物】API仕様書 (API Specification)
│  ├─ 作成者: APIエンジニア
│  ├─ レビュー: フロントエンドチーム、テックリード
│  └─ 承認: テックリード
│
└─ 【成果物】追加ADR (アーキテクチャ決定時)
   └─ 作成者: アーキテクト、テックリード

Week 5: レビュー・承認
└─ 【成果物】設計レビュー記録 (Design Review)
   ├─ 作成者: レビュー主催者
   ├─ 参加者: 全レビュアー
   └─ 承認: テックリード、エンジニアリングマネージャー
```

## 役割別責任マトリクス (RACI)

### 各成果物の役割分担

| 成果物 | テックリード | アーキテクト | エンジニア | PM | QA | セキュリティ |
|--------|:------------:|:------------:|:----------:|:--:|:--:|:------------:|
| **技術提案書** | A/R | R | C | I | I | C |
| **設計書** | A | R | R | I | C | C |
| **ADR** | A | R | C | I | - | C |
| **API仕様書** | A | C | R | I | C | - |
| **設計レビュー記録** | A | R | I | I | R | R |
| **セキュリティ評価書** | A | C | I | I | C | R |

**RACI凡例**:
- **R** (Responsible): 実行責任者 - ドキュメント作成を担当
- **A** (Accountable): 説明責任者 - 最終的な承認者
- **C** (Consulted): 相談先 - レビュー・フィードバック提供
- **I** (Informed): 報告先 - 完成を知らせる

## 成果物の相互関係

### ドキュメント間の依存関係

```
技術提案書 (Technical Proposal)
   │
   ├─ 承認取得
   │
   ▼
設計書 (Design Document)
   │
   ├─ 詳細化
   ├─ 参照 → ADR (重要な決定)
   │         │
   │         ├─ 技術選定の理由
   │         ├─ アーキテクチャパターンの選択
   │         └─ トレードオフの記録
   │
   ├─ 詳細化 → API仕様書
   │           │
   │           └─ エンドポイント定義
   │
   └─ レビュー → 設計レビュー記録
               │
               ├─ フィードバック
               ├─ アクションアイテム
               └─ 承認記録
```

### 情報の流れ

```yaml
information_flow:
  phase_1_proposal:
    input:
      - ビジネス要件
      - 技術調査結果
    output:
      - 技術提案書
      - 初期ADR
    
  phase_2_design:
    input:
      - 承認済み技術提案書
      - 詳細要件
    output:
      - 設計書
      - API仕様書
      - 追加ADR
    
  phase_3_review:
    input:
      - 設計書
      - API仕様書
      - ADR
    output:
      - 設計レビュー記録
      - 修正版ドキュメント
    
  phase_4_implementation:
    input:
      - 承認済み設計書
      - API仕様書
    output:
      - 実装コード
      - テストコード
```

## よくある質問 (FAQ)

### Q1: 小規模な機能追加でも技術提案書は必要ですか?

**A**: プロジェクト規模に応じて判断します:
- **小規模（~1週間）**: 不要。設計書の簡易版で十分
- **中規模（1週間~1ヶ月）**: 簡易版の技術提案を推奨
- **大規模（1ヶ月~）**: 完全な技術提案書が必要

### Q2: ADRはいつ作成すべきですか?

**A**: 以下のような重要な技術決定時に作成します:
- 新しいフレームワーク・ライブラリの採用
- アーキテクチャパターンの選択
- データベースやクラウドサービスの選定
- セキュリティ方式の決定
- 重要なトレードオフを伴う決定

### Q3: API仕様書と設計書の違いは?

**A**:
- **API仕様書**: API契約に特化。エンドポイント、リクエスト・レスポンス形式を定義
- **設計書**: システム全体の設計。アーキテクチャ、データモデル、実装詳細を含む
- API仕様書は設計書のサブセットとも言えますが、API利用者向けに独立して作成を推奨

### Q4: 設計レビューは必須ですか?

**A**: プロジェクト規模による:
- **小規模**: ピアレビューで十分
- **中規模**: 設計レビュー会議を推奨
- **大規模**: 設計レビュー会議が必須

### Q5: ドキュメントは誰が更新しますか?

**A**:
- **所有者**: ドキュメントオーナー（通常は作成者）が更新責任を持つ
- **更新タイミング**: 実装中の大きな変更、本番リリース後の確定
- **レビュー**: 四半期ごとに定期レビューし、必要に応じて更新

```
ステップ1: プロジェクト規模の判断
├─ 期間: [X週間/ヶ月]
├─ チームサイズ: [X人]
└─ 複雑度: [低/中/高]

├─ 設計レビュー会議（必要に応じて）
├─ フィードバック対応
└─ 最終承認

ステップ6: 実装開始
└─ 承認済みドキュメントを参照しながら実装
```
