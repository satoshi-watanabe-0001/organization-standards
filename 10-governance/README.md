# Governance / ガバナンス

## 📋 概要

このディレクトリには、技術ガバナンスに関するドキュメントを集約しています。技術標準の管理、変更プロセス、例外承認、技術レーダーなど、組織の技術的意思決定とガバナンスに関する重要な情報を提供します。

---

## 🎯 目的

### 主な目的
- **技術標準の維持**: 一貫性のある技術スタックの管理
- **変更管理**: 標準の更新プロセスの明確化
- **意思決定の透明性**: 技術選定の基準と理由の文書化
- **例外管理**: 標準からの逸脱を適切に管理
- **継続的改善**: 技術スタックの定期的な見直し
- **AI作業の安全性**: AIエージェントによる作業の適切な制御

### 対象読者
- エンジニアリングリーダー
- アーキテクト
- シニアエンジニア
- プロダクトマネージャー
- 技術選定を行うすべてのメンバー
- AIエージェント（Devin等）を利用するすべてのメンバー

## 💡 よくある質問 (FAQ)

### Q1: 標準に従わなければならないのはいつですか？
**A**: 原則としてすべての新規開発は標準に従います。例外が必要な場合は例外承認プロセスに従ってください。

### Q2: 新しい技術を試したい場合はどうすればよいですか？
**A**: まず小規模なPOC（概念実証）を実施し、その結果をもとにRFCを提出してください。Technology Radarの「Assess」カテゴリーへの追加を検討します。

### Q3: 既存のレガシーコードはどうすればよいですか？
**A**: 段階的な移行計画を立ててください。非推奨ポリシーに従い、優先度を決めて計画的にリファクタリングします。

### Q4: 標準更新の提案はどのように行いますか？
**A**: Standards Update Processに従ってRFCを提出してください。提案には背景、メリット、移行計画を含めます。

### Q5: 緊急時の例外はどうしますか？
**A**: P0/P1インシデントの場合、事後承認も可能です。ただし、インシデント解決後1週間以内に正式な例外申請を提出してください。

### Q6: AIエージェントにCI設定を変更させたい場合は？（新規追加）
**A**: 明示的にファイル名と変更内容を指示してください。例: ".github/workflows/test.ymlのNode.jsバージョンを18に更新してください"。曖昧な指示（"ビルドを高速化してください"等）では、CI設定は変更されません。

### Q7: AIタスクで禁止事項に抵触してしまった場合は？（新規追加）
**A**: レビュー時に指摘し、修正を行ってください。繰り返される場合は、チーム内でAI利用ガイドラインの再確認を実施します。
