# エスカレーション基準の具体化提案 - エグゼクティブサマリー

## 🎯 提案の目的

現在のAI-MASTER-WORKFLOW-GUIDEとチェックリストに記載されているエスカレーション基準を具体化し、AIが自律的かつ一貫性のある判断を行えるようにする。

## 📊 現状の課題

### 確認された4つの主要問題

1. **定量的閾値の欠如**
   - 「重大な影響」「大きく逸脱」などの表現が主観的
   - 数値的な判断基準が存在しない

2. **判断プロセスの不明確さ**
   - Step 0.2「開発に影響するか?」の判断基準が曖昧
   - 「重要 vs 軽微」の区分基準が未定義

3. **ドキュメント間の不整合**
   - GUIDE版: 即座6項目 + 実装後4項目
   - チェックリスト版: 即座4項目 + 実装後3項目

4. **段階的エスカレーションパスの不在**
   - 現状: 「即座」「実装後」の2段階のみ
   - 中間的な判断選択肢がない

## 💡 提案する6つのソリューション

### 提案1: エスカレーション判断マトリクス

**影響度 (Impact)** × **緊急度 (Urgency)** の2軸マトリクス

#### 影響度 (4段階)
- **Critical (致命的)**: セキュリティ・データ整合性・システム全体
- **High (高)**: 複数モジュール・主要機能・アーキテクチャ
- **Medium (中)**: 単一モジュール・限定的機能
- **Low (低)**: UI改善・ログ・ドキュメント

#### 緊急度 (4段階)
- **Immediate (即座)**: 次Stepに進む前に解決必須
- **Before Implementation (実装前)**: Phase 2開始前に解決
- **Before Review (レビュー前)**: Phase 4前に確認
- **After Deployment (デプロイ後)**: Phase 6以降で対応可能

#### 判定結果 (4アクション)
- 🔴 **即座に停止・エスカレーション**: 作業停止、人間判断待機
- 🟠 **実装前エスカレーション**: 確認リクエスト作成、Phase 2前に回答
- 🟡 **実装後レビュー要求**: Phase 4で重点レビュー
- 🟢 **記録して進行**: 標準的想定で実装、判断理由を記録

### 提案2: カテゴリ別の具体的閾値

**6カテゴリ × 3段階** の詳細な判断基準

| カテゴリ | 主要な閾値例 |
|----------|--------------|
| **セキュリティ** | 個人情報取扱 → Critical<br>認証方式変更 → Critical<br>CORS設定 → High |
| **アーキテクチャ** | 3つ以上のサービスに影響 → Critical<br>DB スキーマ変更 → High<br>単一サービス内 → Medium |
| **パフォーマンス** | 50%以上の性能劣化 → Critical<br>20-50%の性能劣化 → High<br>20%未満 → Medium |
| **ビジネスロジック** | 金銭・税金計算 → Critical<br>承認フロー → High<br>バリデーション追加 → Medium |
| **組織標準遵守** | 未承認技術の使用 → Critical<br>規約の例外適用 → High<br>規約範囲内 → Low |
| **工数・スケジュール** | 見積の2倍以上 → Critical<br>見積の50-100%増 → High<br>30%未満増 → Medium |

### 提案3: Step 0.2 不明点処理フローの詳細化

現状の2ステップフローを **4ステップの構造化プロセス** に拡張:

```
不明点を発見
  ↓
[ステップ1] 影響度レベルを判定 (Critical/High/Medium/Low)
  ↓
[ステップ2] 緊急度レベルを判定 (Immediate/Before Impl/Before Review/After Deploy)
  ↓
[ステップ3] マトリクスで対応方針を決定 (🔴/🟠/🟡/🟢)
  ↓
[ステップ4] 判断理由を記録（影響度・緊急度・対応方針・想定した標準対応）
```

### 提案4: 自己診断チェックリスト

AIが機械的に判断できる **5セクション × 質問形式** のチェックリスト

#### セクション構成
- **セクションA**: セキュリティ・コンプライアンス（5問）
  - **1つでも該当 → 🔴 Critical + Immediate**
  
- **セクションB**: アーキテクチャ・技術選定（6問）
  - **2つ以上該当 → 🔴 即座にエスカレーション**
  - **1つ該当 → 🟠 実装前エスカレーション**

- **セクションC**: ビジネスロジック・データ整合性（6問）
  - **2つ以上該当 → 🟠 実装前エスカレーション**
  - **1つ該当 → 🟡 実装後レビュー要求**

- **セクションD**: 組織標準・プロセス遵守（5問）
  - **1つでも該当 → 判定を1段階引き上げ**

- **セクションE**: UI・ログ・ドキュメント（5問）
  - **3つ以上該当 & A/B/C/D項目0個 → 🟢 記録して進行**

### 提案5: ドキュメント間の整合性確保

**マスターソース方式**:
- AI-MASTER-WORKFLOW-GUIDE.md を正として定義
- 各チェックリストはGUIDEへの参照形式に変更
- 新規ガイド `ESCALATION-CRITERIA-GUIDE.md` に詳細を集約

### 提案6: 判定事例集

**6つのサンプル事例** を提供（拡充可能）:

1. **MFAの要否が不明** → 🔴 即座にエスカレーション
2. **API形式の選択（JSON/XML）** → 🟢 記録して進行（標準明記あり）
3. **パフォーマンス要件「高速」** → 🟠 実装前エスカレーション（標準不明）
4. **ログレベルの選択** → 🟢 記録して進行
5. **DBインデックス追加** → 🟡 実装後レビュー要求
6. **UI文言の変更** → 🟢 記録して進行
