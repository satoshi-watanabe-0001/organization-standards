## 📚 各Phaseの特徴

### Phase 0: 要件分析・企画（15KB、459行）
- **7ステップ**: PBI読み込み→情報抽出→技術要件整理→タスク分解→リスク分析→要件書作成→確認依頼
- **PBIタイプ別アプローチ**: 7タイプ（新規プロジェクト、機能追加、バグ修正等）
- **参照ドキュメント**: 9-reference、10-governance、08-templates

### Phase 1: プロジェクト初期化（16KB、480行）
- **6ステップ**: 環境確認→リポジトリ初期化→プロジェクト構造作成→開発ツール設定→CI/CD基本設定→初期ドキュメント作成
- **コアドキュメント**: Git Workflow、CI/CD Pipeline、技術スタック、ツール設定
- **スキップ可能性**: 既存プロジェクトへの機能追加時は簡易版

### Phase 2: 設計（18KB、459行）
- **6ステップ**: アーキテクチャ設計→データモデル設計→API設計→セキュリティ設計→パフォーマンス設計→設計ドキュメント作成
- **コアドキュメント**: 設計原則、API Architecture（294KB超大規模）、データベース設計、セキュリティアーキテクチャ
- **プロジェクトタイプ別**: フロントエンド/バックエンド/マイクロサービス/クラウドネイティブ

### Phase 3: 実装（28KB、663行）
- **タスク別ガイド**: タスク粒度別（レベル0-4）、タスクタイプ別、アーキテクチャレイヤー別、CRUD操作別
- **重要度**: 🔴必須、🟡推奨、⚪参考の3段階
- **既に軽量型**: リネームとメタデータ更新のみ

### Phase 4: レビュー・品質保証（18KB、476行）
- **7ステップ**: コードレビュー→テスト戦略確認→ユニットテスト→統合テスト→E2Eテスト→コード品質チェック→パフォーマンス・セキュリティテスト
- **コアドキュメント**: コードレビュー標準、テスト戦略、ユニット・統合・E2Eテスト標準
- **カバレッジ目標**: ユニットテスト80%以上

### Phase 5: デプロイメント（14KB、429行）
- **4ステップ**: デプロイ前チェック→デプロイ実行→デプロイ後検証→リリースノート作成
- **デプロイ戦略**: ブルー/グリーン、カナリア、ローリングデプロイメント
- **コアドキュメント**: リリース管理、CI/CDパイプライン、変更管理

### Phase 6: 運用・保守（15KB、442行）
- **6ステップ**: モニタリング設定→ログ管理設定→日常モニタリング→インシデント対応→変更管理→技術的負債管理
- **継続的フェーズ**: 終わりがなく、Phase 0に戻って改善サイクルを回す
- **コアドキュメント**: モニタリング・ログ、インシデント管理、変更管理、技術的負債管理

## 🎯 ガイドの役割分担

### 既存ドキュメント（01-10フォルダ）
- **What**: 何を守るべきか（標準、基準、ベストプラクティス）
- **Why**: なぜ守るべきか（原則、理由）
- **タイプ**: 参照型・辞書・リファレンス

### Phaseガイド（phase-X-xxx-guide.md）
- **When**: いつ参照すべきか（フェーズごとのタイミング）
- **How**: どのように使うか（実行手順）
- **Which**: どのドキュメントを参照すべきか（ナビゲーション）
- **タイプ**: 実行型・手順書
