# PBIタイプ別テスト要件チェックリスト

> PBIのタイプに応じた適切なテスト要件の確認

**目的**: PBIタイプごとに必要なテストが実施されていることを確認  
**使用タイミング**: Phase 3（実装）のテスト実施時、Phase 4（レビュー・QA）の品質確認時  
**参照元**: [testing-standards.md](../../03-development-process/testing-standards.md)

---

## ✅ 使い方

このチェックリストは、実装するPBIのタイプに応じて、必要なテストを選択・実施するために使用します。

### チェック方法
1. PBIタイプを特定（以下の7タイプのいずれか）
2. 該当するタイプのセクションを確認
3. 必須テストをすべて実施
4. 推奨テストを可能な限り実施

### PBIタイプの7分類
1. 新規プロジェクト
2. 既存システムへの機能追加
3. バグ修正
4. リファクタリング
5. 設計変更
6. Hotfix
7. 実験的機能（PoC）

---

## 📋 タイプ1: 新規プロジェクト

### 必須テスト（Phase 3完了前に実施）

- [ ] **ユニットテスト**: すべての新規関数・メソッドに対して実施
  - カバレッジ目標: 80%以上
  - すべてのエッジケース、例外処理をカバー

- [ ] **統合テスト**: モジュール間の連携を確認
  - API連携、データフロー、外部サービス統合をテスト

- [ ] **E2Eテスト**: 主要なユーザーシナリオを確認
  - 最低3つの主要シナリオをカバー

- [ ] **パフォーマンステスト**: レスポンスタイム、スループットを測定
  - 非機能要件を満たすことを確認

### 推奨テスト（リソースが許せば実施）

- [ ] **負荷テスト**: 想定負荷での動作確認
- [ ] **セキュリティテスト**: 脆弱性スキャン、ペネトレーションテスト
- [ ] **アクセシビリティテスト**: WCAG準拠確認（UI がある場合）
- [ ] **互換性テスト**: 複数ブラウザ・デバイスでの動作確認

---

## 📋 タイプ2: 既存システムへの機能追加

### 必須テスト（Phase 3完了前に実施）

- [ ] **ユニットテスト**: 新規追加した関数・メソッドのテスト
  - カバレッジ目標: 新規コードの80%以上

- [ ] **統合テスト**: 既存機能との連携を確認
  - 新規機能が既存機能に影響しないことを確認

- [ ] **リグレッションテスト**: 既存機能が正常動作することを確認
  - 既存の自動テストスイートを実行
  - 主要な既存機能を手動確認

- [ ] **E2Eテスト**: 新機能を含むシナリオをテスト
  - 最低2つのシナリオをカバー

### 推奨テスト（リソースが許せば実施）

- [ ] **パフォーマンステスト**: 新機能追加による性能影響を確認
- [ ] **データ整合性テスト**: 既存データへの影響確認

---

## 📋 タイプ3: バグ修正

### 必須テスト（Phase 3完了前に実施）

- [ ] **再現テスト**: バグが再現することを確認
  - 修正前に必ず実施

- [ ] **修正検証テスト**: バグが修正されたことを確認
  - 報告された不具合が解消していることを確認

- [ ] **ユニットテスト**: 修正箇所に対するテストを追加
  - 同じバグが再発しないようテストケースを追加

- [ ] **リグレッションテスト**: 修正が他に影響していないことを確認
  - 関連する既存機能を重点的にテスト

### 推奨テスト（リソースが許せば実施）

- [ ] **類似バグの探索**: 同様の問題が他にないか確認
- [ ] **境界値テスト**: バグ修正箇所の境界条件を確認

---

## 📋 タイプ4: リファクタリング

### 必須テスト（Phase 3完了前に実施）

- [ ] **リグレッションテスト**: 既存機能が変わらないことを確認
  - **最重要**: すべての既存テストが合格することを確認

- [ ] **ユニットテスト**: リファクタリング箇所のテストを実行
  - カバレッジが低下していないことを確認

- [ ] **統合テスト**: モジュール間の連携が変わらないことを確認

### 推奨テスト（リソースが許せば実施）

- [ ] **パフォーマンステスト**: 性能が改善または維持されていることを確認
- [ ] **コード品質チェック**: 静的解析ツールで品質向上を確認

---

## 📋 タイプ5: 設計変更

### 必須テスト（Phase 3完了前に実施）

- [ ] **ユニットテスト**: 変更された全モジュールをテスト
  - カバレッジ目標: 80%以上

- [ ] **統合テスト**: 新しいアーキテクチャでの連携を確認
  - データフロー、API契約が正しく動作することを確認

- [ ] **リグレッションテスト**: 既存機能が正常動作することを確認
  - **広範囲**: 影響範囲が大きいため、すべての既存機能を確認

- [ ] **マイグレーションテスト**: データ移行が正しく行われることを確認（該当する場合）

### 推奨テスト（リソースが許せば実施）

- [ ] **パフォーマンステスト**: 設計変更による性能影響を確認
- [ ] **負荷テスト**: 新アーキテクチャでの耐久性確認

---

## 📋 タイプ6: Hotfix

### 必須テスト（Phase 3完了前に実施）

- [ ] **再現テスト**: 障害が再現することを確認
  - 本番環境と同等の環境で実施

- [ ] **修正検証テスト**: 障害が修正されたことを確認
  - 報告された障害が解消していることを確認

- [ ] **最小限のリグレッションテスト**: 修正が他に影響していないことを確認
  - **時間制約**: 影響範囲を特定し、最小限のテストを実施
  - 直接関連する機能のみを確認

### 推奨テスト（可能な範囲で実施）

- [ ] **本番環境シミュレーション**: 本番と同じ条件で動作確認
- [ ] **ロールバックテスト**: 必要に応じて切り戻しが可能か確認

---

## 📋 タイプ7: 実験的機能（PoC）

### 必須テスト（Phase 3完了前に実施）

- [ ] **基本動作テスト**: 主要な機能が動作することを確認
  - **最小限**: ハッピーパスのみでも可

- [ ] **統合テスト**: 既存システムとの基本的な連携を確認
  - 既存システムに影響しないことを確認

### 推奨テスト（リソースが許せば実施）

- [ ] **ユニットテスト**: 主要な関数のテスト
  - カバレッジ目標: 50%程度でも可
- [ ] **パフォーマンステスト**: 実現可能性の検証

### 注意事項

⚠️ **実験的機能は本番投入前に通常の品質基準を満たすこと**
- PoCから本番機能に昇格する場合は、タイプ1またはタイプ2の基準を適用
