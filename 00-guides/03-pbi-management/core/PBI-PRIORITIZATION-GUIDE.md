## はじめに

### このガイドの目的

Product Backlogの効果的な管理において、**優先順位付け（Prioritization）**は最も重要な活動の一つです。このガイドは、以下を提供します：

- **体系的な優先順位付け手法**: 複数の評価手法とその適用方法
- **PBIタイプ別戦略**: 各PBIタイプに応じた優先順位判断基準
- **実践的な実例**: 具体的なケーススタディと判断プロセス
- **JIRA連携**: 優先順位管理のツール活用方法
- **AI支援**: 自動化可能な優先順位付けロジック

### 優先順位付けの重要性

適切な優先順位付けにより、以下のメリットが得られます：

- ✅ **ビジネス価値の最大化**: 最も価値の高い機能を優先的に提供
- ✅ **リスクの早期軽減**: 重要なリスクに早期に対処
- ✅ **リソースの最適配分**: 限られたリソースを効果的に活用
- ✅ **ステークホルダーの満足度向上**: 透明性のある意思決定プロセス
- ✅ **開発チームの生産性向上**: 明確な方向性と優先順位

### 対象読者

- **Product Owner**: バックログの優先順位決定の責任者
- **開発チーム**: 優先順位の理解と技術的評価の提供
- **プロジェクトマネージャー**: リソース計画と進捗管理
- **自律的AIエージェント**: 優先順位の自動提案と分析

## 優先順位付けの基本原則

優先順位付けは、以下の4つの基本原則に基づいて行います：

### 1. ビジネス価値の評価

#### 収益インパクト

**直接的な収益効果**:
- 新規顧客獲得による売上増加
- 既存顧客の単価向上（アップセル・クロスセル）
- チャーン率（解約率）の低減
- コスト削減（運用コスト、サポートコスト等）

**評価方法**:
```
収益インパクトスコア = 
  予想収益増加額（年間） ÷ 100万円

例: 年間500万円の収益増加が見込まれる場合
  → スコア = 5
```

#### ユーザー満足度

**ユーザー価値の評価**:
- ユーザーの課題解決度（Must-have vs Nice-to-have）
- ユーザー体験の改善度（UXスコア）
- 顧客要望の件数・頻度
- NPS（Net Promoter Score）への影響

**評価基準**:
| レベル | 説明 | スコア |
|--------|------|--------|
| **Critical** | ユーザーが製品を使えない致命的な課題 | 10 |
| **High** | ユーザーの主要業務に大きく影響 | 7-9 |
| **Medium** | ユーザー体験を改善、業務効率化 | 4-6 |
| **Low** | あれば便利だが必須ではない | 1-3 |

#### 戦略的重要性

**長期的な戦略への貢献**:
- 市場競争力の強化
- 新市場への参入機会
- 技術的優位性の確保
- ブランド価値の向上

**評価のポイント**:
- 経営戦略との整合性
- ロードマップ上の位置づけ
- 競合他社との差別化要素
- 市場トレンドへの適合性

---

### 2. 技術的複雑度の評価

#### 実装難易度

**技術的な複雑さの要因**:
- 新技術の学習コスト
- 既存システムとの統合複雑度
- データ移行の難易度
- パフォーマンス要件の厳しさ

**評価スケール（Tシャツサイズ）**:
| サイズ | 説明 | 想定工数 | スコア |
|--------|------|----------|--------|
| **XS** | 非常に簡単、既存パターンの適用 | 1-2日 | 1 |
| **S** | 簡単、若干の新規実装 | 3-5日 | 2 |
| **M** | 中程度、複数コンポーネントの変更 | 1-2週 | 3 |
| **L** | 複雑、アーキテクチャ変更を伴う | 2-4週 | 5 |
| **XL** | 非常に複雑、大規模なリファクタリング | 1-2ヶ月 | 8 |

#### 技術的リスク

**リスク要因の評価**:
- 技術的不確実性（未知の技術領域）
- 依存ライブラリの安定性
- パフォーマンス懸念
- セキュリティリスク
- スケーラビリティの課題

**リスク評価マトリックス**:
```
リスクスコア = 発生確率（%） × 影響度（1-10）

例: 発生確率50%、影響度8の場合
  → リスクスコア = 0.5 × 8 = 4.0
```

#### 依存関係

**依存関係の種類**:
- **技術的依存**: 他のPBIの実装完了が必要
- **データ依存**: 特定のデータが必要
- **外部依存**: 外部システムやAPIの対応待ち
- **リソース依存**: 特定のスキルを持つメンバーが必要

**依存関係の管理**:
- 依存関係の可視化（JIRA Linkage）
- ブロッカーの早期解決
- 並行開発可能な部分の特定

---

### 3. リスク評価

#### ビジネスリスク

**ビジネスへの悪影響**:
- 顧客喪失のリスク
- 売上減少のリスク
- ブランド毀損のリスク
- 法的・コンプライアンスリスク

**評価項目**:
| リスク種別 | 評価基準 | 対応優先度 |
|-----------|---------|-----------|
| **顧客流出** | 既存顧客の解約見込み件数 | 最高 |
| **売上損失** | 遅延による機会損失額 | 高 |
| **法的リスク** | 法令違反の可能性 | 最高 |
| **評判リスク** | SNS等での炎上リスク | 中-高 |

#### 技術リスク

**技術的な問題発生リスク**:
- システム障害のリスク
- パフォーマンス劣化のリスク
- データ損失のリスク
- セキュリティ脆弱性のリスク

**技術リスクの評価**:
```
技術リスク優先度 = 
  Severity（重大度）× Probability（発生確率）

Severity: 1(Low) - 10(Critical)
Probability: 0.1(10%) - 1.0(100%)
```

#### タイムリスク

**時間的制約とリスク**:
- 市場投入タイミングの逸失
- 契約上の期限
- 季節要因（繁忙期等）
- 競合他社の動向

**時間的緊急性の評価**:
- **今すぐ**: 24時間以内の対応必須（HOT）
- **今週中**: 1週間以内の対応必須
- **今月中**: 1ヶ月以内の対応推奨
- **今四半期中**: 3ヶ月以内の対応
- **いつでも**: 特定の期限なし

---

### 4. ROI（投資対効果）分析

#### ROIの計算式

```
ROI = (期待リターン - 投資コスト) ÷ 投資コスト × 100%

期待リターン: ビジネス価値（収益増加、コスト削減等）
投資コスト: 開発工数 × 人件費単価 + その他コスト
```

#### 具体例

**例1: 新機能開発（NFD）**
```
期待リターン:
  - 年間収益増加: 600万円
  - 3年間の累計: 1,800万円

投資コスト:
  - 開発工数: 3人月 × 100万円/人月 = 300万円
  - インフラコスト: 50万円/年 × 3年 = 150万円
  - 合計: 450万円

ROI = (1,800万円 - 450万円) ÷ 450万円 × 100%
    = 300%

→ 投資に対して3倍のリターン（優先度: 高）
```

**例2: 技術的改善（REF）**
```
期待リターン:
  - 開発効率20%向上
  - 年間節約工数: 2人月 × 100万円 = 200万円
  - 3年間の累計: 600万円

投資コスト:
  - リファクタリング工数: 1人月 = 100万円

ROI = (600万円 - 100万円) ÷ 100万円 × 100%
    = 500%

→ 投資に対して5倍のリターン（優先度: 高）
```

#### ROIに基づく優先順位付け

| ROI範囲 | 優先度 | 判断 |
|---------|--------|------|
| **500%以上** | 最高 | 最優先で実施 |
| **200-500%** | 高 | 積極的に実施 |
| **100-200%** | 中 | 条件次第で実施 |
| **50-100%** | 低 | 慎重に検討 |
| **50%未満** | 最低 | 実施を見送り |

## 優先順位付け手法

複数の優先順位付け手法を理解し、状況に応じて適切な手法を選択します。

### 手法一覧

| 手法名 | 適用場面 | 複雑度 | 推奨度 |
|--------|---------|--------|--------|
| **MoSCoW法** | 初期段階、シンプルな分類 | 低 | ★★★★★ |
| **RICE評価** | 定量的評価が必要な場合 | 中 | ★★★★☆ |
| **WSJF** | SAFe/アジャイル環境 | 中 | ★★★★☆ |
| **Kano分析** | 顧客満足度重視 | 高 | ★★★☆☆ |
| **Value vs Effort** | 視覚的な優先順位付け | 低 | ★★★★★ |

### 3.1 MoSCoW法

#### 概要

MoSCoW法は、PBIを4つのカテゴリに分類する最もシンプルで直感的な手法です。

#### 4つのカテゴリ

##### M - Must have（必須）
**定義**: プロジェクトの成功に不可欠な機能
**特徴**:
- これがないとシステムが機能しない
- 法的・契約上の義務
- ビジネスクリティカル

**判定基準**:
- 「これがないと、プロダクトをリリースできるか？」 → No なら Must
- 「これがないと、法的問題が発生するか？」 → Yes なら Must
- 「これがないと、ビジネスが回らないか？」 → Yes なら Must

**例**:
- ログイン機能（認証システム）
- 決済機能（ECサイト）
- データバックアップ機能
- 個人情報保護対応（GDPR等）

##### S - Should have（重要）
**定義**: 重要だが、必須ではない機能
**特徴**:
- ユーザー体験を大きく改善
- ビジネス価値が高い
- 回避策が存在する

**判定基準**:
- 「これがないと、ユーザーは不便を感じるか？」 → Yes なら Should
- 「これがないと、競合に劣るか？」 → Yes なら Should
- 「一時的な回避策で対応可能か？」 → Yes なら Should

**例**:
- パスワードリセット機能（サポートで対応可能）
- 高度な検索フィルター
- ダッシュボード機能
- プッシュ通知

##### C - Could have（あれば良い）
**定義**: あれば嬉しいが、なくても問題ない機能
**特徴**:
- ユーザー体験を若干改善
- ビジネスインパクトは限定的
- 実装が簡単

**判定基準**:
- 「これがなくても、誰も困らないか？」 → Yes なら Could
- 「余裕があれば実装したいか？」 → Yes なら Could

**例**:
- ダークモード
- カスタムテーマ
- ショートカットキー
- アニメーション効果

##### W - Won't have（今回は対象外）
**定義**: 今回のリリースでは実装しない機能
**特徴**:
- 将来的には検討する可能性あり
- 現時点では優先度が低い
- リソースが不足している

**判定基準**:
- 「今回のリリースで必要か？」 → No なら Won't
- 「将来のバージョンに延期できるか？」 → Yes なら Won't

**例**:
- 多言語対応（現時点で日本市場のみ）
- AIレコメンド機能（データ蓄積が必要）
- 高度な分析機能

#### MoSCoW法の適用プロセス

**ステップ1: 初期分類**
```
全PBIをリストアップ
↓
各PBIを仮に4カテゴリに分類
↓
チームでレビュー
```

**ステップ2: Must haveの絞り込み**
```
Must haveが全体の60%以上の場合
↓
「本当にMustか?」を再検討
↓
Should haveへの降格を検討
```

**推奨比率**:
- Must have: 30-40%
- Should have: 30-40%
- Could have: 20-30%
- Won't have: 残り

**ステップ3: ステークホルダー合意**
```
Product Ownerと分類を確認
↓
開発チームの実現可能性を確認
↓
最終的なMoSCoW分類を決定
```

#### 判定フローチャート

```
PBIを評価
    ↓
Q1: これがないとシステムが機能しないか？
    Yes → Must have
    No → Q2へ
    ↓
Q2: ユーザーが大きな不便を感じるか?
    Yes → Should have
    No → Q3へ
    ↓
Q3: 実装が簡単で、あると嬉しいか?
    Yes → Could have
    No → Won't have
```

#### 実例: ECサイトのPBI分類

| PBI | カテゴリ | 理由 |
|-----|---------|------|
| 商品一覧表示 | **M** | コア機能、これがないとサイトが成立しない |
| カート機能 | **M** | 購入に必須 |
| クレジットカード決済 | **M** | 決済手段が必要 |
| 商品レビュー機能 | **S** | ユーザー体験向上、競合も実装 |
| お気に入り機能 | **S** | ユーザー利便性向上 |
| ソーシャルログイン | **C** | あれば便利だが必須ではない |
| ダークモード | **C** | ユーザー層によっては不要 |
| AIレコメンド | **W** | 将来実装、データ蓄積が先 |

### 3.2 RICE評価

#### 概要

RICE評価は、4つの要素を数値化してスコアを算出する定量的な手法です。

#### 4つの評価要素

##### R - Reach（到達範囲）
**定義**: 一定期間内に影響を受けるユーザー数

**測定単位**:
- **ユーザー数**: 月間アクティブユーザー（MAU）
- **トランザクション数**: 月間処理件数
- **アクセス数**: 月間ページビュー

**例**:
```
月間アクティブユーザー: 10,000人
この機能を使用する割合: 50%
→ Reach = 5,000人/月
```

##### I - Impact（影響度）
**定義**: 個々のユーザーへの影響の大きさ

**評価スケール**:
| スコア | 影響度 | 説明 |
|--------|--------|------|
| **3** | Massive | ユーザー体験を劇的に改善 |
| **2** | High | 大きく改善 |
| **1** | Medium | 適度に改善 |
| **0.5** | Low | 若干改善 |
| **0.25** | Minimal | 最小限の改善 |

**判断基準**:
- **Massive (3)**: 製品の使いやすさが劇的に向上、コンバージョン率大幅UP
- **High (2)**: 主要な課題を解決、満足度が明確に向上
- **Medium (1)**: ユーザー体験が改善、若干の満足度向上
- **Low (0.5)**: 一部のユーザーに便利、限定的な改善
- **Minimal (0.25)**: ほとんど気づかれない程度の改善

##### C - Confidence（確信度）
**定義**: 見積もりの確実性

**評価スケール**:
| スコア | 確信度 | 説明 |
|--------|--------|------|
| **100%** | High | データに基づく確実な見積もり |
| **80%** | Medium | ある程度のデータと経験に基づく |
| **50%** | Low | 推測が多い、不確実性が高い |

**判断基準**:
- **100%**: ユーザー調査データあり、過去の類似事例あり
- **80%**: 部分的なデータあり、チームの経験に基づく
- **50%**: データが少ない、新しい試み

##### E - Effort（工数）
**定義**: 開発コスト（人月）

**測定単位**:
- **人月**: 1人のメンバーが1ヶ月働く工数
- **ストーリーポイント**: チームの見積もり単位

**例**:
```
開発: 15日（0.5人月）
テスト: 5日（0.17人月）
デプロイ: 2日（0.07人月）
→ 合計 Effort = 0.74人月 ≈ 0.75人月
```

**工数見積もりのヒント**:
- 実装のみでなく、テスト・デプロイも含める
- レビュー時間も考慮
- 過去の類似PBIを参考にする

#### RICE スコアの計算

**計算式**:
```
RICE Score = (Reach × Impact × Confidence) ÷ Effort
```

**具体例**:

**例1: 新規登録フローの簡素化（ENH）**
```
Reach: 2,000人/月（新規登録者数）
Impact: 2（High - コンバージョン率20%改善見込み）
Confidence: 80%（A/Bテストデータあり）
Effort: 0.5人月

RICE Score = (2,000 × 2 × 0.8) ÷ 0.5
           = 3,200 ÷ 0.5
           = 6,400
```

**例2: ダークモード追加（ENH）**
```
Reach: 5,000人/月（利用見込み）
Impact: 0.5（Low - 一部ユーザーの利便性向上）
Confidence: 80%（ユーザーリクエストあり）
Effort: 1.0人月

RICE Score = (5,000 × 0.5 × 0.8) ÷ 1.0
           = 2,000 ÷ 1.0
           = 2,000
```

**結果**: 新規登録フロー簡素化の方が優先度が高い（6,400 > 2,000）

#### RICE評価の適用プロセス

**ステップ1: データ収集**
```
- ユーザー数の確認（Google Analytics等）
- 過去の工数データの収集
- ユーザーフィードバックの分析
```

**ステップ2: チームでスコアリング**
```
Product Owner: Reach, Impactを評価
開発チーム: Effortを見積もり
両者で: Confidenceを合意
```

**ステップ3: RICE スコア計算**
```
Excelまたはスプレッドシートで一括計算
↓
スコアの高い順にソート
↓
上位から優先的に実装
```

**ステップ4: 定期的な見直し**
```
月次または四半期ごとに再評価
↓
新しいデータで更新
↓
優先順位を調整
```

#### RICE評価のベストプラクティス

**✅ Do（推奨）**:
- チーム全体でスコアリングを行う
- データに基づいて評価する
- 定期的に見直す
- スコアだけでなく、戦略的要素も考慮

**❌ Don't（非推奨）**:
- 一人で決めない
- 直感だけで評価しない
- 一度決めたら変更しない
- スコアを絶対視しすぎない

### 3.3 WSJF (Weighted Shortest Job First)

#### 概要

WSJF（Weighted Shortest Job First）は、SAFe（Scaled Agile Framework）で推奨される優先順位付け手法です。**遅延コスト（Cost of Delay）**を重視し、最も経済的価値の高いPBIを優先します。

#### WSJFの計算式

```
WSJF = Cost of Delay（CoD）÷ Job Size

Cost of Delay = 
  User-Business Value（ユーザー・ビジネス価値）
  + Time Criticality（時間的緊急性）
  + Risk Reduction / Opportunity Enablement（リスク低減・機会創出）
```

#### Cost of Delay（CoD）の3要素

##### 1. User-Business Value（ユーザー・ビジネス価値）
**定義**: ユーザーとビジネスへの価値

**評価基準（1-10）**:
| スコア | 説明 |
|--------|------|
| **10** | ビジネスクリティカル、収益に直結 |
| **7-9** | 高いビジネス価値、ユーザー満足度向上 |
| **4-6** | 中程度の価値、改善効果あり |
| **1-3** | 限定的な価値、一部ユーザーのみ |

**例**:
- 決済機能の高速化 → 10（コンバージョン率に直結）
- レポート機能の追加 → 6（業務効率化）
- UI の微調整 → 2（限定的な改善）

##### 2. Time Criticality（時間的緊急性）
**定義**: 時間の経過による価値の減少

**評価基準（1-10）**:
| スコア | 説明 |
|--------|------|
| **10** | 今すぐ必要、遅延で価値が大幅に減少 |
| **7-9** | 早期リリースが重要、遅延で機会損失 |
| **4-6** | タイミングは重要だが多少の遅延は許容 |
| **1-3** | いつでも良い、時間的制約なし |

**例**:
- クリスマス商戦向け機能 → 10（期限厳守）
- 法改正対応 → 10（期限あり）
- 新機能追加 → 5（早いほうが良いが柔軟）
- 内部リファクタリング → 2（時間制約なし）

##### 3. Risk Reduction / Opportunity Enablement（リスク低減・機会創出）
**定義**: リスクの軽減または新しい機会の創出

**評価基準（1-10）**:
| スコア | 説明 |
|--------|------|
| **10** | 重大なリスク回避、大きな機会創出 |
| **7-9** | 重要なリスク軽減、新市場参入 |
| **4-6** | 中程度のリスク低減、将来の布石 |
| **1-3** | 限定的なリスク対応、小さな機会 |

**例**:
- セキュリティ脆弱性修正 → 10（重大リスク回避）
- 新技術スタックへの移行 → 8（技術的負債削減）
- 新API公開 → 7（エコシステム拡大）
- コードコメント追加 → 2（保守性向上）

#### Job Size（実装サイズ）

**定義**: 実装に必要な工数

**評価基準（フィボナッチ数列）**:
| サイズ | 説明 | 工数目安 |
|--------|------|----------|
| **1** | 極小 | 1-2日 |
| **2** | 小 | 3-5日 |
| **3** | 小-中 | 1週間 |
| **5** | 中 | 2週間 |
| **8** | 大 | 3-4週間 |
| **13** | 極大 | 1-2ヶ月 |

**注意**: あまりに大きい（13以上）場合は、PBI分割を検討

#### WSJF計算の具体例

**例1: 決済フローの高速化（ENH）**
```
User-Business Value: 9（コンバージョン率改善）
Time Criticality: 7（競合対策で早期実施が望ましい）
Risk Reduction: 5（ユーザー離脱リスク低減）

Cost of Delay = 9 + 7 + 5 = 21

Job Size: 5（2週間）

WSJF = 21 ÷ 5 = 4.2
```

**例2: ダークモード追加（ENH）**
```
User-Business Value: 3（一部ユーザーの利便性向上）
Time Criticality: 2（時間的制約なし）
Risk Reduction: 1（リスク・機会への影響小）

Cost of Delay = 3 + 2 + 1 = 6

Job Size: 3（1週間）

WSJF = 6 ÷ 3 = 2.0
```

**例3: セキュリティ脆弱性修正（BUG）**
```
User-Business Value: 10（ユーザーデータ保護）
Time Criticality: 10（即座対応必須）
Risk Reduction: 10（重大リスク回避）

Cost of Delay = 10 + 10 + 10 = 30

Job Size: 2（3-5日）

WSJF = 30 ÷ 2 = 15.0
```

**優先順位**: セキュリティ修正（15.0）> 決済高速化（4.2）> ダークモード（2.0）

#### WSJF適用のプロセス

**ステップ1: チームでスコアリング**
```
Product Owner、開発チーム、ステークホルダーが参加
↓
各PBIについて、CoD 3要素をディスカッション
↓
プランニングポーカーで合意形成
```

**ステップ2: Job Sizeの見積もり**
```
開発チームが技術的複雑度を評価
↓
過去の類似PBIを参考に見積もり
↓
フィボナッチ数列でサイズを決定
```

**ステップ3: WSJF計算とソート**
```
スプレッドシートで一括計算
↓
WSJFスコアの高い順にソート
↓
上位から実装順序を決定
```

**ステップ4: 定期的な再評価**
```
スプリント計画ごとに見直し
↓
ビジネス環境の変化を反映
↓
優先順位を調整
```

#### WSJFのメリット・デメリット

**✅ メリット**:
- 経済的価値を重視した優先順位付け
- 時間的要素を考慮（遅延コスト）
- チーム全体での合意形成
- SAFeフレームワークとの統合

**⚠️ デメリット**:
- スコアリングに時間がかかる
- 主観的な評価が入りやすい
- 小規模チームには過剰な場合も

#### WSJFのベストプラクティス

**推奨事項**:
- 定期的なキャリブレーション（基準の調整）
- 過去のスコアと実績の比較
- スコアだけでなく戦略的判断も重視
- 透明性のあるプロセス

### 3.4 Kano分析

#### 概要

Kano分析は、**顧客満足度**の観点からPBIを分類する手法です。同じ機能でも、顧客満足への影響は異なります。

#### Kanoモデルの5つの品質カテゴリ

##### 1. 当たり前品質（Must-be Quality）
**定義**: あって当然の機能、ないと不満

**特徴**:
- 実装しても満足度は上がらない
- 実装しないと大きな不満が発生
- **最優先で実装が必要**

**例**:
- ログイン機能
- データ保存機能
- エラーメッセージ表示
- 基本的なセキュリティ

**優先順位**: **最高**（必須機能）

##### 2. 一元的品質（One-dimensional Quality）
**定義**: 満足度と比例する機能

**特徴**:
- 実装すると満足度が上がる
- 実装しないと不満が生じる
- **充実度が満足度に直結**

**例**:
- 検索機能の精度
- ページ読み込み速度
- 機能の豊富さ
- サポート対応の質

**優先順位**: **高**（満足度向上に直結）

##### 3. 魅力的品質（Attractive Quality）
**定義**: 驚きと満足をもたらす機能

**特徴**:
- 実装すると大きな満足が得られる
- 実装しなくても不満は生じない
- **差別化要素**

**例**:
- AIレコメンド機能
- パーソナライゼーション
- 革新的なUI/UX
- ゲーミフィケーション

**優先順位**: **中-高**（差別化のため）

##### 4. 無関心品質（Indifferent Quality）
**定義**: あってもなくても満足度に影響しない

**特徴**:
- ユーザーは気にしない
- 実装しても評価されない
- **リソースの無駄**

**例**:
- 使われない高度な機能
- 過剰な設定オプション
- 不要な通知

**優先順位**: **最低**（実装しない）

##### 5. 逆品質（Reverse Quality）
**定義**: あると不満が生じる機能

**特徴**:
- 実装すると満足度が下がる
- ユーザーが望まない機能
- **実装すべきでない**

**例**:
- 過度な広告
- 強制的なチュートリアル
- 不要な個人情報要求
- 複雑すぎるUI

**優先順位**: **実装禁止**

#### Kano分析の実施方法

**ステップ1: ユーザーアンケート作成**

各機能について、2つの質問をする：

**Q1（機能的質問）**: この機能があったら、どう感じますか？
- A. 嬉しい
- B. 当然だ
- C. どちらでもない
- D. 仕方ない
- E. 嫌だ

**Q2（非機能的質問）**: この機能がなかったら、どう感じますか？
- A. 嬉しい
- B. 当然だ
- C. どちらでもない
- D. 仕方ない
- E. 嫌だ

**ステップ2: Kanoカテゴリ判定表**

| Q1→<br>Q2↓ | A. 嬉しい | B. 当然 | C. どちらでも | D. 仕方ない | E. 嫌だ |
|-----------|---------|--------|------------|-----------|-------|
| **A. 嬉しい** | 疑問 | 魅力的 | 魅力的 | 魅力的 | 一元的 |
| **B. 当然** | 逆品質 | 無関心 | 無関心 | 無関心 | 当たり前 |
| **C. どちらでも** | 逆品質 | 無関心 | 無関心 | 無関心 | 当たり前 |
| **D. 仕方ない** | 逆品質 | 無関心 | 無関心 | 無関心 | 当たり前 |
| **E. 嫌だ** | 逆品質 | 逆品質 | 逆品質 | 逆品質 | 疑問 |

**ステップ3: 結果集計と分析**

```
各カテゴリの回答数を集計
↓
最も多いカテゴリを採用
↓
優先順位を決定
```

#### Kano分析の具体例

**例: ECサイトの機能評価**

| 機能 | Kanoカテゴリ | 優先順位 | 実装判断 |
|------|------------|---------|---------|
| 商品検索 | **当たり前** | 最高 | 必須実装 |
| 決済機能 | **当たり前** | 最高 | 必須実装 |
| 配送状況追跡 | **一元的** | 高 | 実装推奨 |
| レビュー機能 | **一元的** | 高 | 実装推奨 |
| AIレコメンド | **魅力的** | 中 | 差別化のため実装 |
| バーチャル試着 | **魅力的** | 中 | リソース次第 |
| 過度なアニメーション | **逆品質** | - | 実装しない |
| 複雑な設定画面 | **無関心** | 最低 | 実装しない |

#### Kano分析のベストプラクティス

**✅ Do（推奨）**:
- 定期的にユーザー調査を実施
- カテゴリは時間とともに変化（魅力的→一元的→当たり前）
- 複数のセグメント別に分析
- 競合他社の動向も考慮

**❌ Don't（非推奨）**:
- 一度の調査結果を絶対視しない
- ユーザーの声だけで決めない（技術的判断も必要）
- 少数の回答で判断しない

### 3.5 Value vs Effort マトリックス

#### 概要

Value vs Effort マトリックスは、**ビジネス価値**と**実装工数**の2軸でPBIを視覚的に分類する最もシンプルで効果的な手法です。

#### 2×2マトリックス

```
    高
    ↑
価  │  Major Projects  │  Quick Wins
値  │  (大規模施策)     │  (優先実施)
    │  計画的実施       │  最優先
    │─────────────┼─────────────
    │  Time Sinks      │  Fill-ins
    │  (時間の無駄)     │  (すきま対応)
    │  回避            │  低優先
    ↓─────────────┴─────────────→
    低                              高
              工数（Effort）
```

#### 4つの象限の詳細

##### 象限1: Quick Wins（高価値・低工数）
**特徴**:
- **高いビジネス価値**を**少ない工数**で実現
- **最優先で実装**すべき「お宝」
- ROIが最も高い

**戦略**:
- 即座に実装開始
- 次スプリントに組み込む
- 早期リリースで価値を回収

**例**:
- 明らかなバグ修正（ユーザー影響大）
- 簡単なUI改善（コンバージョン率改善）
- 既存機能の簡単な拡張
- パフォーマンス改善（設定変更のみ）

##### 象限2: Major Projects（高価値・高工数）
**特徴**:
- **高いビジネス価値**だが**多くの工数**が必要
- 計画的な実施が必要
- 複数スプリントに分割

**戦略**:
- 詳細な計画を立てる
- PBI分割して段階的に実装
- MVPから開始して段階的に拡張
- リソース確保を事前に調整

**例**:
- 新しい主要機能の開発
- アーキテクチャの大規模改善
- 新プラットフォームへの対応
- データ移行を伴う機能改修

##### 象限3: Fill-ins（低価値・低工数）
**特徴**:
- **低いビジネス価値**で**少ない工数**
- すきま時間に対応
- 急ぐ必要はない

**戦略**:
- スプリントに余裕があれば実施
- 新メンバーの練習課題として活用
- バックログの最下位に配置

**例**:
- 小さなUI調整
- ログメッセージの改善
- 内部ツールの小改善
- ドキュメント更新

##### 象限4: Time Sinks（低価値・高工数）
**特徴**:
- **低いビジネス価値**に**多くの工数**が必要
- ROIが極めて低い
- **実装を避けるべき**

**戦略**:
- 実装しない
- 代替案を検討
- 要件を再検討（本当に必要か？）
- ステークホルダーと価値を再議論

**例**:
- 使われない高度な機能
- 過度に複雑な設定オプション
- ニッチなユースケースへの対応
- 過剰な最適化

#### Value vs Effort マトリックスの作成手順

**ステップ1: 価値の評価**

各PBIのビジネス価値をスコアリング（1-10）：

```
評価項目:
- 収益インパクト（+5点）
- ユーザー満足度（+3点）
- 戦略的重要性（+2点）

合計: 10点満点
```

**価値スコアの例**:
| PBI | 収益 | 満足度 | 戦略 | 合計 |
|-----|------|--------|------|------|
| 決済フロー改善 | 5 | 3 | 2 | **10** |
| レビュー機能 | 2 | 3 | 1 | **6** |
| ダークモード | 0 | 2 | 0 | **2** |

**ステップ2: 工数の評価**

各PBIの実装工数をスコアリング（1-10）：

```
1-2: 1-3日（XS）
3-4: 4-7日（S）
5-6: 1-2週（M）
7-8: 3-4週（L）
9-10: 1ヶ月以上（XL）
```

**ステップ3: マトリックスへのプロット**

```
スプレッドシートまたはホワイトボードに2軸を描画
↓
各PBIをプロット
↓
4象限に分類
↓
象限ごとに優先順位を決定
```

**ステップ4: 優先順位の決定**

```
優先順位:
1. Quick Wins（高価値・低工数）
2. Major Projects（高価値・高工数）
3. Fill-ins（低価値・低工数）
4. Time Sinks（低価値・高工数）← 実施しない
```

#### 具体例: ECサイトの機能評価

| PBI | 価値 | 工数 | 象限 | 優先順位 |
|-----|------|------|------|---------|
| 決済エラー修正 | 10 | 2 | **Quick Wins** | 1 |
| カート放棄通知 | 8 | 3 | **Quick Wins** | 2 |
| 新規決済手段追加 | 9 | 8 | **Major Projects** | 3 |
| レコメンド機能 | 7 | 7 | **Major Projects** | 4 |
| ログ改善 | 2 | 2 | **Fill-ins** | 5 |
| 詳細な分析機能 | 3 | 9 | **Time Sinks** | 実施しない |

#### Value vs Effort マトリックスのメリット

**✅ メリット**:
- **視覚的で直感的**: 誰でも理解しやすい
- **迅速な意思決定**: 短時間で優先順位付け可能
- **合意形成が容易**: ステークホルダー間の議論がしやすい
- **柔軟性が高い**: 評価基準をカスタマイズ可能

**⚠️ 注意点**:
- 主観的な評価になりがち → データで補完
- リスク要素が考慮されない → 別途リスク評価が必要
- 長期的視点が欠ける → 戦略との整合性確認

#### Value vs Effort マトリックスのベストプラクティス

**推奨事項**:
- チーム全体でマトリックスを作成（ワークショップ形式）
- 定期的に見直し（四半期ごと）
- データで裏付けられた評価を行う
- Time Sinks は実施しない勇気を持つ
- Quick Wins を見逃さない

## PBIタイプ別優先順位戦略

8つのPBIタイプごとに、優先順位付けのポイントと戦略を説明します。

### タイプ別優先順位マトリックス

| PBIタイプ | デフォルト優先度 | 主要評価軸 | 推奨手法 |
|----------|--------------|----------|---------|
| **HOT** | 最高 | ビジネスインパクト、緊急性 | 即座判断 |
| **BUG** | 高-中 | Severity、影響範囲 | Severity評価 |
| **NFD** | 中-高 | ビジネス価値、ROI | RICE/WSJF |
| **ENH** | 中 | ユーザー価値、工数 | Value vs Effort |
| **ARC** | 中-低 | 技術的負債、長期効果 | ROI分析 |
| **REF** | 低-中 | 開発効率、保守性 | 技術的ROI |
| **POC** | 低 | 学習価値、リスク | タイムボックス |
| **NPI** | 最高 | 戦略的重要性 | 経営判断 |

### HOT（緊急修正）の優先順位戦略

#### デフォルト優先度: **最高**

#### 特徴
- **24時間以内の対応**が必要
- ビジネスへの影響が甚大
- 通常のプロセスを省略

#### 優先順位判定基準

**最優先（即座対応）**:
- [ ] システムダウン、データ損失
- [ ] セキュリティ脆弱性の公開
- [ ] 法的問題の発生リスク
- [ ] 大規模なユーザー影響（全ユーザー）

**高優先（24時間以内）**:
- [ ] 主要機能の停止
- [ ] 重要な顧客への影響
- [ ] データ不整合
- [ ] パフォーマンス重大劣化

**中優先（48時間以内）**:
- [ ] 一部機能の停止
- [ ] 特定ユーザーへの影響
- [ ] 回避策が存在する問題

#### 評価プロセス

**ステップ1: 影響範囲の評価**
```
影響ユーザー数:
- 全ユーザー → 最優先
- 50%以上 → 高優先
- 10-50% → 中優先
- 10%未満 → 低優先（HOTではなくBUGへ）
```

**ステップ2: ビジネスインパクト評価**
```
収益影響:
- 100万円/日以上 → 最優先
- 50-100万円/日 → 高優先
- 10-50万円/日 → 中優先
- 10万円/日未満 → 低優先（BUGへ）
```

**ステップ3: 暫定対応 vs 恒久対応**
```
暫定対応（Workaround）:
- 2-4時間で対応
- 機能を一時停止、または簡易修正
- 恒久対応を別PBIとして起票

恒久対応（Permanent Fix）:
- 根本原因の解決
- 通常のBUGとして計画的に対応
```

#### HOT優先順位の実例

**例1: クレジットカード決済エラー（全ユーザー影響）**
```
影響範囲: 全ユーザー
ビジネスインパクト: 売上停止（推定損失: 500万円/日）
評価結果: 最優先（即座対応）

対応:
- 暫定対応: 決済サービスを別プロバイダーに切替（2時間）
- 恒久対応: 根本原因調査と修正（別PBIで計画）
```

**例2: 特定ブラウザでのUI崩れ**
```
影響範囲: 5%のユーザー（特定ブラウザのみ）
ビジネスインパクト: 限定的
評価結果: HOTではなく、通常のBUGとして対応

理由: 影響範囲が限定的、回避策あり（別ブラウザ使用）
```

### BUG（バグ修正）の優先順位戦略

#### デフォルト優先度: **高-中**

#### 特徴
- 既存機能の不具合
- Severity（重大度）に基づく優先順位付け

#### Severity（重大度）評価

##### Critical（致命的）
**定義**: システムの主要機能が使用不可

**特徴**:
- データ損失・破損
- セキュリティ脆弱性
- 全ユーザーへの影響
- 回避策なし

**優先順位**: **最高（即座対応）**

**例**:
- ログインができない
- データが保存されない
- 決済が完了しない
- SQLインジェクション脆弱性

##### High（高）
**定義**: 主要機能に重大な影響

**特徴**:
- 主要ユースケースが実行困難
- 多数のユーザーへの影響
- 簡易な回避策が存在

**優先順位**: **高（1週間以内）**

**例**:
- 検索結果が不正確
- レポート生成に失敗
- 特定条件下でエラー発生
- パフォーマンス大幅劣化

##### Medium（中）
**定義**: 機能の一部に影響

**特徴**:
- 一部ユーザーへの影響
- 回避策が存在
- ユーザー体験の低下

**優先順位**: **中（2-4週間以内）**

**例**:
- UI表示の乱れ
- ソート機能の不具合
- 通知が届かない
- エラーメッセージ不適切

##### Low（低）
**定義**: 軽微な問題

**特徴**:
- 限定的な影響
- 容易な回避策あり
- ユーザー体験への影響小

**優先順位**: **低（バックログ管理）**

**例**:
- 表示の微妙なずれ
- 不要なログ出力
- ドキュメントの誤記
- デッドコード

#### 影響範囲の評価

| 影響範囲 | 評価 | 優先度への影響 |
|---------|------|--------------|
| **全ユーザー** | 最大 | +2段階 |
| **50%以上** | 大 | +1段階 |
| **10-50%** | 中 | 変更なし |
| **10%未満** | 小 | -1段階 |

#### 回避策の有無

| 回避策 | 優先度への影響 |
|--------|-------------|
| **回避策なし** | +1段階 |
| **複雑な回避策** | 変更なし |
| **簡単な回避策** | -1段階 |

#### BUG優先順位の実例

**例1: ログイン失敗（全ユーザー）**
```
Severity: Critical
影響範囲: 全ユーザー
回避策: なし

優先順位: 最高（即座対応、HOTレベル）
```

**例2: 検索結果の不正確さ**
```
Severity: High
影響範囲: 全ユーザー
回避策: 別の検索方法あり

優先順位: 高（1週間以内）
```

**例3: 特定ブラウザでのUI崩れ**
```
Severity: Medium
影響範囲: 5%のユーザー
回避策: 別ブラウザ使用

優先順位: 中（2-4週間以内）
```

---

### NFD（新規機能開発）/ ENH（既存機能改修）の優先順位戦略

#### デフォルト優先度: **中-高**

#### 特徴
- ビジネス価値が主要な判断基準
- ROI、RICE、WSJFなど定量的評価が有効

#### 推奨評価手法

**1. RICE評価**（定量的評価が可能な場合）
```
Reach: ユーザー数（Google Analyticsから取得）
Impact: A/Bテスト結果、ユーザー調査
Confidence: データの信頼性
Effort: 過去の類似PBIから見積もり

→ RICE Scoreで優先順位決定
```

**2. WSJF評価**（SAFe環境の場合）
```
User-Business Value: ステークホルダー評価
Time Criticality: 市場タイミング
Risk Reduction: 競合分析
Job Size: 開発チーム見積もり

→ WSJFスコアで優先順位決定
```

**3. Value vs Effort マトリックス**（迅速な判断が必要な場合）
```
Value: ビジネス価値（1-10）
Effort: 実装工数（1-10）

→ Quick Winsを最優先
```

#### NFD/ENH優先順位の実例

**例1: 新規決済手段追加（NFD）**
```
RICE評価:
- Reach: 10,000人/月（想定利用者）
- Impact: 2（コンバージョン率10%改善）
- Confidence: 80%（市場調査あり）
- Effort: 2人月

RICE Score = (10,000 × 2 × 0.8) ÷ 2 = 8,000

優先順位: 高
```

**例2: ダッシュボード改善（ENH）**
```
RICE評価:
- Reach: 500人/月（限定ユーザー）
- Impact: 1（業務効率若干改善）
- Confidence: 80%
- Effort: 1人月

RICE Score = (500 × 1 × 0.8) ÷ 1 = 400

優先順位: 中-低
```

**結果**: 新規決済手段を優先（8,000 > 400）

---

### ARC（アーキテクチャ改善）/ REF（リファクタリング）の優先順位戦略

#### デフォルト優先度: **中-低**

#### 特徴
- 技術的負債の解消
- 短期的な価値は見えにくい
- 長期的な開発効率への影響大

#### 優先順位判定基準

**高優先（即座対応）**:
- [ ] セキュリティリスクの解消
- [ ] パフォーマンス問題の根本解決
- [ ] 新機能開発のブロッカー
- [ ] サポート終了ライブラリの更新

**中優先（計画的実施）**:
- [ ] コード品質の改善（保守性向上）
- [ ] テストカバレッジの向上
- [ ] ドキュメント整備
- [ ] アーキテクチャの最適化

**低優先（バックログ管理）**:
- [ ] コードスタイルの統一
- [ ] 不要なコードの削除
- [ ] 小規模なリファクタリング

#### 技術的ROIの計算

```
技術的ROI = 
  （開発効率改善による節約工数 × 期間）÷ 投資工数

例: リファクタリング
- 投資工数: 1人月
- 開発効率改善: 20%
- チームサイズ: 5人
- 計算期間: 1年（12ヶ月）

節約工数 = 5人 × 12ヶ月 × 20% = 12人月
技術的ROI = 12人月 ÷ 1人月 = 12倍

→ 投資に対して12倍のリターン（高優先度）
```

#### ビジネス側との価値共有

**技術的負債を可視化**:
- 開発速度の低下グラフ
- バグ発生率の推移
- 新機能開発にかかる時間の増加

**ビジネス言語で説明**:
```
技術的な表現:
「コードが複雑化しているため、リファクタリングが必要」

ビジネス言語:
「現在の状態では、新機能開発に通常の2倍の時間がかかっています。
1ヶ月のリファクタリングで、今後1年間の開発速度が20%向上し、
年間で12人月（約1,200万円）のコスト削減が見込めます」
```

#### ARC/REF優先順位の実例

**例1: レガシーコードのリファクタリング**
```
技術的ROI:
- 投資工数: 2人月
- 開発効率改善: 30%
- チームサイズ: 10人
- 計算期間: 1年

節約工数 = 10人 × 12ヶ月 × 30% = 36人月
技術的ROI = 36 ÷ 2 = 18倍

優先順位: 高（ROI高い）
```

**例2: コードスタイル統一**
```
技術的ROI:
- 投資工数: 0.5人月
- 開発効率改善: 5%
- チームサイズ: 5人
- 計算期間: 1年

節約工数 = 5人 × 12ヶ月 × 5% = 3人月
技術的ROI = 3 ÷ 0.5 = 6倍

優先順位: 中（ROI中程度）
```

#### 20%ルール

**技術的負債対策の定期枠確保**:
```
各スプリントのキャパシティの20%を技術的改善に割り当て

例: スプリントキャパシティ 100ポイント
- ビジネス機能開発: 80ポイント
- 技術的改善（ARC/REF）: 20ポイント

→ 技術的負債の蓄積を防ぐ
```

---

### POC（実験的機能）の優先順位戦略

#### デフォルト優先度: **低**

#### 特徴
- 技術検証、実験、概念実証
- タイムボックスの設定が重要
- 学習価値を重視

#### 優先順位判定基準

**高優先（積極的に実施）**:
- [ ] 新技術の評価（将来の主要技術）
- [ ] 技術的リスクの検証（大規模プロジェクト前）
- [ ] ユーザー受容性のテスト（革新的機能）
- [ ] パフォーマンス検証（スケーラビリティ）

**中優先（リソース次第）**:
- [ ] 技術的好奇心による実験
- [ ] 将来的な機能の試作
- [ ] サードパーティツールの評価

**低優先（見送り）**:
- [ ] 明確な目的のない実験
- [ ] 過度に長期のPOC
- [ ] ビジネス価値が不明確

#### タイムボックスの設定

```
POCの期間制限:
- 短期: 2-3日（技術調査）
- 中期: 1週間（プロトタイプ作成）
- 長期: 2週間（包括的検証）

最大: 2週間まで
（それ以上は通常のNFDとして計画）
```

#### Go/No-Go判断基準

POC完了後、以下の基準で判断：

**Go（本開発へ移行）**:
- 技術的実現可能性が確認された
- ユーザー価値が明確
- ビジネスケースが成立
- リスクが許容範囲内

**No-Go（中止）**:
- 技術的に実現困難
- ユーザー価値が低い
- コストが高すぎる
- 代替案がより優れている

#### POC優先順位の実例

**例1: 新しいAI技術の評価**
```
目的: 次世代レコメンド機能の実現可能性
期間: 1週間
学習価値: 高（将来の主要機能）

優先順位: 高（戦略的重要性）
```

**例2: 新しいUIライブラリの評価**
```
目的: 開発効率向上の可能性
期間: 3日
学習価値: 中（開発効率改善）

優先順位: 中（リソース次第）
```

---

### NPI（新規プロジェクト立ち上げ）の優先順位戦略

#### デフォルト優先度: **最高**

#### 特徴
- 経営戦略レベルの意思決定
- 全Phaseを実施（0→1→2→3→4→5→6）
- 大規模なリソース投資

#### 優先順位判定基準

**最優先（経営判断）**:
- [ ] 経営戦略との直接的な整合性
- [ ] 市場機会の獲得
- [ ] 競合対策の必要性
- [ ] 法的・契約的義務

**評価プロセス**:
```
ステップ1: 経営層による戦略評価
↓
ステップ2: 市場分析・競合分析
↓
ステップ3: ROI試算（3-5年スパン）
↓
ステップ4: リソース確保計画
↓
ステップ5: Go/No-Go判断
```

#### NPIの評価は本ガイドの範囲外

NPIは通常のPBI優先順位付けとは異なり、**経営戦略レベルの意思決定**が必要です。

詳細は以下を参照：
- **[AI-MASTER-WORKFLOW-GUIDE.md](../../02-ai-guides/AI-MASTER-WORKFLOW-GUIDE.md)**: Phase 0-6 の詳細
- **[PBI-CREATION-BEST-PRACTICES.md](../core/PBI-CREATION-BEST-PRACTICES.md)**: NPIの作成方法

## ステークホルダー管理と合意形成

優先順位付けは、複数のステークホルダー間の合意形成が不可欠です。

### 主要ステークホルダー

| ステークホルダー | 役割 | 優先順位への関心 |
|---------------|------|---------------|
| **Product Owner** | バックログの責任者 | ビジネス価値の最大化 |
| **開発チーム** | 実装の担当 | 技術的実現可能性、工数 |
| **ビジネスオーナー** | ビジネス目標の責任者 | ROI、市場タイミング |
| **ユーザー** | 製品の利用者 | ユーザー体験、課題解決 |
| **経営層** | 戦略的意思決定 | 戦略との整合性、投資対効果 |

---

### Product Ownerとの調整

#### Product Ownerの責任

- バックログの優先順位決定
- ビジネス価値の評価
- ステークホルダー間の調整
- 最終意思決定

#### 調整プロセス

**ステップ1: ビジネス優先度のヒアリング**
```
Product Ownerからビジネス目標をヒアリング
↓
各PBIがビジネス目標にどう貢献するか確認
↓
MoSCoW分類またはRICE評価を実施
```

**ステップ2: 技術的制約の説明**
```
開発チームから技術的複雑度を説明
↓
依存関係やリスクを共有
↓
実現可能なスケジュールを提示
```

**ステップ3: 優先順位の合意**
```
Product Ownerが優先順位を決定
↓
開発チームが実現可能性を確認
↓
最終的なバックログ順序を確定
```

---

### 開発チームとの合意

#### 開発チームの役割

- 技術的複雑度の評価
- 工数見積もり
- リスクの特定
- 実装方法の提案

#### 合意形成のポイント

**技術的複雑度の共有**:
```
複雑なPBIについて:
- なぜ複雑なのか説明
- リスクを明確化
- 代替案を提示
```

**キャパシティの確認**:
```
スプリント計画時:
- チームのベロシティを確認
- 余裕を持ったコミットメント
- 優先順位に基づく選択
```

**技術的負債の主張**:
```
ARC/REFの重要性を説明:
- 現状の課題を可視化
- 将来への影響を説明
- 技術的ROIを提示
```

---

### 優先順位の合意プロセス

#### バックログリファインメント会議

**目的**: バックログの優先順位を定期的に見直す

**頻度**: 週次（スプリントの中間）

**参加者**:
- Product Owner（必須）
- 開発チーム（必須）
- ステークホルダー（任意）

**アジェンダ**:
```
1. 新規PBIの追加とレビュー（15分）
2. 優先順位の見直し（20分）
3. 上位PBIの詳細化（20分）
4. 工数見積もり（15分）
5. 次スプリント計画の準備（10分）
```

**成果物**:
- 優先順位付けされたバックログ
- 詳細化されたPBI（上位10-20個）
- 工数見積もり

#### プランニングポーカー

**目的**: チーム全体で工数見積もりの合意

**手法**:
```
ステップ1: PBIの説明（Product Owner）
↓
ステップ2: 質疑応答
↓
ステップ3: 各メンバーがカードで見積もり
   （フィボナッチ数列: 1, 2, 3, 5, 8, 13...）
↓
ステップ4: 全員同時にカード公開
↓
ステップ5: 最大値・最小値の理由を説明
↓
ステップ6: 再度見積もり
↓
ステップ7: 合意形成
```

**メリット**:
- チーム全員の視点を反映
- 楽観的・悲観的見積もりのバランス
- 透明性の高いプロセス

#### コンセンサス形成

**合意形成の原則**:
- **透明性**: 判断基準を明確に
- **データ駆動**: 客観的なデータに基づく
- **対話**: 異なる意見を尊重
- **柔軟性**: 状況に応じて調整

**意見が対立した場合**:
```
ステップ1: 各意見の根拠を明確化
↓
ステップ2: データで検証可能な部分を特定
↓
ステップ3: 小規模実験（POC、A/Bテスト）で検証
↓
ステップ4: 結果に基づいて再議論
↓
ステップ5: Product Ownerが最終判断
```

## 優先順位付けの実例集

実際のプロジェクトでの優先順位付け例を示します。

### 実例1: ECサイト機能開発（RICE評価適用）

#### 背景
既存のECサイトに新機能を追加する際の優先順位付け

#### 評価対象PBI

**PBI-1: レビュー機能追加（NFD）**
```
Reach: 8,000人/月（全ユーザーの80%が利用見込み）
Impact: 2（購入意思決定に大きく影響、コンバージョン率10%改善見込み）
Confidence: 80%（競合分析と顧客調査に基づく）
Effort: 1.5人月

RICE Score = (8,000 × 2 × 0.8) ÷ 1.5
           = 12,800 ÷ 1.5
           = 8,533
```

**PBI-2: ソーシャルログイン追加（ENH）**
```
Reach: 3,000人/月（新規登録者の推定利用率）
Impact: 1（登録ハードルの若干低減）
Confidence: 80%（一般的な効果データあり）
Effort: 0.5人月

RICE Score = (3,000 × 1 × 0.8) ÷ 0.5
           = 2,400 ÷ 0.5
           = 4,800
```

**PBI-3: AIレコメンド機能（NFD）**
```
Reach: 10,000人/月（全ユーザー）
Impact: 1.5（購入単価15%向上見込み）
Confidence: 50%（新技術で不確実性高い）
Effort: 3.0人月

RICE Score = (10,000 × 1.5 × 0.5) ÷ 3.0
           = 7,500 ÷ 3.0
           = 2,500
```

**PBI-4: ダークモード対応（ENH）**
```
Reach: 2,000人/月（推定利用者）
Impact: 0.5（一部ユーザーの利便性向上）
Confidence: 80%（ユーザー要望あり）
Effort: 0.75人月

RICE Score = (2,000 × 0.5 × 0.8) ÷ 0.75
           = 800 ÷ 0.75
           = 1,067
```

#### 優先順位決定

| 順位 | PBI | RICE Score | 判断 |
|------|-----|-----------|------|
| **1** | レビュー機能 | 8,533 | 最優先実施（高スコア） |
| **2** | ソーシャルログイン | 4,800 | 次スプリントで実施 |
| **3** | AIレコメンド | 2,500 | リソース次第で実施 |
| **4** | ダークモード | 1,067 | 低優先、バックログ管理 |

#### Before / After 比較

**Before（直感的な判断）**:
```
1. AIレコメンド（「最新技術で魅力的」）
2. レビュー機能
3. ソーシャルログイン
4. ダークモード
```

**After（RICE評価）**:
```
1. レビュー機能（最高ROI）
2. ソーシャルログイン（低工数で効果あり）
3. AIレコメンド（工数とリスクを考慮して後回し）
4. ダークモード（影響範囲が限定的）
```

**効果**:
- データに基づく客観的な判断
- ステークホルダー間の合意形成が容易
- ROI重視の順序付け

### 実例4: 緊急対応の判断（HOT vs BUG）

#### シナリオ
本番環境で問題発生、HOTかBUGかの判定

#### ケース1: 決済エラー（全ユーザー影響）

**状況**:
```
現象: クレジットカード決済が全て失敗
発生時刻: 13:00（営業時間中）
影響: 全ユーザー（決済不可）
推定損失: 500万円/日
```

**判定**:
```
□ 影響範囲: 全ユーザー → ✓
□ ビジネス影響: 売上停止 → ✓
□ 回避策: なし → ✓
□ 緊急性: 即座対応必要 → ✓

結果: HOT（緊急修正）
対応: 全員招集、即座対応
```

**対応フロー**:
```
1. 暫定対応（30分）:
   - 決済サービスプロバイダー切替
   - または決済を一時停止、復旧見込み告知

2. 根本原因調査（2-4時間）:
   - ログ分析
   - 外部APIの状況確認

3. 恒久対応（別PBI）:
   - 複数決済プロバイダー対応
   - フェイルオーバー機構の実装
```

#### ケース2: 特定機能のエラー（一部ユーザー影響）

**状況**:
```
現象: レビュー投稿時にエラー
発生時刻: 深夜2:00
影響: レビュー機能のみ（閲覧は可能）
推定影響: 10%のユーザー
```

**判定**:
```
□ 影響範囲: 一部機能、一部ユーザー → ✗
□ ビジネス影響: 限定的 → ✗
□ 回避策: 後でレビュー投稿可能 → ✗
□ 緊急性: 営業時間外 → ✗

結果: BUG（通常のバグ修正）
対応: 翌営業日に調査、1週間以内に修正
```

**対応フロー**:
```
1. 当日:
   - 監視アラートを確認
   - 簡易調査（ログ確認）
   - 影響範囲の特定

2. 翌営業日:
   - 詳細調査
   - 修正計画の策定
   - Severity評価（High）

3. 1週間以内:
   - 修正とテスト
   - デプロイ
```

#### HOT vs BUG 判定チェックリスト

| 項目 | HOT | BUG |
|------|-----|-----|
| 影響範囲 | 全ユーザーまたは主要機能 | 一部ユーザー・機能 |
| ビジネス影響 | 売上停止、重大損失 | 限定的 |
| 回避策 | なし | あり |
| 対応期限 | 24時間以内 | 1週間以上可 |
| 対応体制 | 全員招集 | 通常体制 |

#### 優先順位関連のカスタムフィールド

##### 1. Priority（標準フィールド）

**設定値**:
- **Highest**: 緊急対応（HOT、Critical BUG）
- **High**: 高優先（高RICE/WSJFスコア）
- **Medium**: 中優先（標準的なPBI）
- **Low**: 低優先（バックログ管理）
- **Lowest**: 最低優先（実施予定なし）

**使用方法**:
```
Settings → Issues → Priority
↓
各優先度に説明を追加:
- Highest: 24時間以内対応、ビジネスクリティカル
- High: 1週間以内対応、重要なビジネス価値
- Medium: 2-4週間、通常のPBI
- Low: 時期未定、バックログ管理
```

##### 2. Business Value（カスタム数値フィールド）

**フィールド設定**:
```
Field Name: Business Value
Field Type: Number
Description: ビジネス価値（1-10）
```

**評価基準**:
| スコア | 説明 |
|--------|------|
| 10 | ビジネスクリティカル、収益に直結 |
| 7-9 | 高いビジネス価値 |
| 4-6 | 中程度の価値 |
| 1-3 | 限定的な価値 |

##### 3. Technical Complexity（カスタム選択フィールド）

**フィールド設定**:
```
Field Name: Technical Complexity
Field Type: Select List (single choice)
Options:
- Very High (XL)
- High (L)
- Medium (M)
- Low (S)
- Very Low (XS)
```

##### 4. RICE Score（カスタム数値フィールド）

**フィールド設定**:
```
Field Name: RICE Score
Field Type: Number (decimal)
Description: (Reach × Impact × Confidence) ÷ Effort
```

**計算方法**:
```
手動計算または
JIRAオートメーションで自動計算
（後述）
```

##### 5. WSJF Score（カスタム数値フィールド）

**フィールド設定**:
```
Field Name: WSJF Score
Field Type: Number (decimal)

```

##### 6. Severity（バグ専用カスタムフィールド）

**フィールド設定**:
```
Field Name: Severity
Field Type: Select List (single choice)
Options:
- Critical
- High
- Medium
- Low
```

**Conditionalフィールド**:
```
Issue Type = Bug の場合のみ表示
```

### 2. データに基づく判断

#### データ収集の重要性

**✅ 収集すべきデータ**:
- ユーザー数、アクセス数（Google Analytics）
- コンバージョン率、離脱率
- ユーザーフィードバック、サポート問い合わせ件数
- 開発速度（ベロシティ）、バグ発生率
- 過去のPBIの工数実績

**❌ 避けるべき判断**:
- 声の大きいステークホルダーの意見のみ
- 直感だけでの判断
- 過去の成功体験への固執
- 確証バイアス（自分の意見を裏付けるデータのみ収集）

#### データの活用例

**例1: ユーザーアンケート結果の活用**
```

機能Bへの要望: 50件

→ 機能Aの優先度を上げる根拠
```

**例2: A/Bテスト結果の活用**
```
パターンA: コンバージョン率 3.2%
パターンB: コンバージョン率 4.1%

→ パターンBの実装を優先
```

**ツールでの実現**:

## よくある失敗パターンと対策

優先順位付けでよくある失敗とその対策を説明します。

### 失敗1: すべてが最優先になる

#### 症状
```
Product Backlog:
- PBI-1: Priority = Highest
- PBI-2: Priority = Highest
- PBI-3: Priority = Highest
- PBI-4: Priority = Highest
- ...（全てHighest）
```

#### 原因
- ステークホルダーが全て「最優先」を主張
- 優先順位付けの基準がない
- トレードオフの意識がない

#### 対策

**対策1: 強制的なランク付け**
```
MoSCoW法の適用:
- Must have: 最大30%まで
- Should have: 30-40%
- Could have: 20-30%
- Won't have: 残り

ルール: Must haveが30%を超えたら再評価必須
```

**対策2: スタックランキング**
```
全PBIを1位から順位付け
- 同じ順位は許可しない
- 必ず優劣をつける

Product Ownerが最終決定権を持つ
```

**対策3: リソース制約の明示**
```
「次スプリントは100ポイントまで。
優先順位順に選択すると、上位10件のみ実施可能です」

→ 自然と優先順位が明確になる
```

---

### 失敗2: 技術的負債が後回しになる

#### 症状
```
Backlog:
- 新機能開発: 大量
- 技術的改善（REF/ARC）: 常に後回し

結果:
- 開発速度の低下
- バグ発生率の増加
- チームのモチベーション低下
```

#### 原因
- ビジネス価値が見えにくい
- 短期的な成果を優先
- 技術的負債の影響を理解されていない

#### 対策

**対策1: 20%ルールの導入**
```
スプリントキャパシティの20%を技術的改善に固定

例: 100ポイントのスプリント
- ビジネス機能: 80ポイント
- 技術的改善: 20ポイント（固定枠）

→ 毎スプリント必ず技術的改善を実施
```

**対策2: 技術的ROIの可視化**
```
技術的負債のコストを数値化:

現状:
- 新機能開発: 通常の2倍の時間
- バグ修正: 週5件発生

リファクタリング後:
- 新機能開発: 通常の時間（50%削減）
- バグ修正: 週1件（80%削減）

年間節約: 36人月（約3,600万円）
投資: 2人月（約200万円）
ROI: 18倍

→ ビジネス言語で説明
```

**対策3: 技術的負債の定期レポート**
```
月次で報告:
- 技術的負債の総量（日数換算）
- 開発速度の推移グラフ
- バグ発生率の推移

→ 経営層・ビジネスオーナーへの可視化
```

### 失敗3: 声の大きいステークホルダーに流される

#### 症状
```
ステークホルダーA（部長）:
「この機能を最優先で！」

Product Owner:
「はい、わかりました」（データ確認せず）

結果:
- 本当に重要なPBIが後回し
- データに基づかない意思決定
- 他のステークホルダーの不満
```

#### 原因
- 客観的な評価基準がない
- ステークホルダーの力関係
- Product Ownerの優柔不断

#### 対策

**対策1: 客観的評価基準の徹底**
```
全PBIをRICE評価:

ステークホルダーAの要望:
- RICE Score: 500

既存の高優先PBI:
- RICE Score: 8,000

→ データで優先順位を示す
「こちらのPBIの方がスコアが高いため、先に実施します」
```

**対策2: バックログリファインメント会議での議論**
```
個別の要望ではなく、会議で全体最適を議論:

1. 全ステークホルダーが参加
2. 各PBIを公平に評価
3. チーム全体で合意形成

→ 一部のステークホルダーの独断を防ぐ
```

**対策3: Product Ownerの権限の明確化**
```
Product Ownerの責任と権限を明確に:
- バックログの優先順位決定権は Product Owner
- ステークホルダーは提案は可能だが、決定権はない
- 最終判断はデータと戦略に基づく

→ 組織全体での合意
```

### 失敗4: 短期的視点のみで判断

#### 症状
```
優先順位:
1. 今月の売上に直結する機能
2. 今月の売上に直結する機能
3. 今月の売上に直結する機能

結果:
- 長期的な競争力の低下
- 技術的負債の蓄積
- 戦略的投資がされない
```

#### 原因
- 四半期・年間の目標のみ評価
- 長期的視点の欠如
- 戦略とバックログの乖離

#### 対策

**対策1: 戦略的ロードマップとの整合性確認**
```
四半期ごとに確認:

戦略的目標:
- 新市場への参入
- 技術的優位性の確保
- ブランド価値の向上

バックログの割り当て:
- 短期的な売上機能: 50%
- 中期的な戦略投資: 30%
- 長期的な技術投資: 20%

→ バランスの取れた投資
```

**対策2: ホライズンプランニング**
```
3つのホライゾンでバックログを管理:

Horizon 1（現在〜3ヶ月）:
- 既存ビジネスの最適化
- バグ修正、小改善

Horizon 2（3ヶ月〜1年）:
- 新機能開発
- 市場拡大

Horizon 3（1年以上）:
- 新規事業
- 革新的技術の導入

→ 各ホライズンに適切にリソース配分
```

**対策3: OKR（Objectives and Key Results）との連携**
```
四半期OKR:
Objective: 新市場でのシェア10%獲得

Key Results:
- KR1: 新市場向け機能のリリース
- KR2: ユーザー獲得 1,000人
- KR3: 顧客満足度 NPS 40以上

バックログ優先順位:
- KR1達成のためのPBIを優先
- 短期的な売上よりOKR達成を重視

→ 戦略との明確な連携
```

```python
def calculate_rice_score(pbi):
    """
    RICE Score = (Reach × Impact × Confidence) / Effort
    """
    reach = pbi.get('reach', 0)
    impact = pbi.get('impact', 1)
    confidence = pbi.get('confidence', 0.5)
    effort = pbi.get('effort', 1)
    
    if effort == 0:
        return 0
    
    rice_score = (reach * impact * confidence) / effort
    return round(rice_score, 2)

def suggest_priority_by_rice(rice_score):
    """
    RICE Scoreに基づく優先度提案
    """
    if rice_score >= 5000:
        return "Highest"
    elif rice_score >= 2000:
        return "High"
    elif rice_score >= 500:
        return "Medium"
    else:
        return "Low"
```

**WSJF Score 計算**:
```python
def calculate_wsjf_score(pbi):
    """
    WSJF = Cost of Delay / Job Size
    CoD = User-Business Value + Time Criticality + Risk Reduction
    """
    user_business_value = pbi.get('user_business_value', 5)
    time_criticality = pbi.get('time_criticality', 5)
    risk_reduction = pbi.get('risk_reduction', 5)
    job_size = pbi.get('job_size', 5)
    
    if job_size == 0:
        return 0
    
    cost_of_delay = user_business_value + time_criticality + risk_reduction
    wsjf_score = cost_of_delay / job_size
    return round(wsjf_score, 2)

def suggest_priority_by_wsjf(wsjf_score):
    """
    WSJF Scoreに基づく優先度提案
    """
    if wsjf_score >= 10:
        return "Highest"
    elif wsjf_score >= 5:
        return "High"
    elif wsjf_score >= 2:
        return "Medium"
    else:
        return "Low"
```

- [ ] **優先順位変更の理由を記録したか？**
  - 変更理由の明記
  - ステークホルダーへの通知
  - JIRAコメントへの記録

- [ ] **技術的負債に対処しているか？**
  - 20%ルールの適用
  - ARC/REFの定期実施
  - 技術的ROIの評価

- [ ] **データで判断を裏付けているか？**
  - ユーザーデータの確認
  - 過去の実績分析
  - A/Bテスト結果の活用

**📌 重要なリマインダー**

優先順位付けは、**継続的なプロセス**です。以下を忘れずに：

1. **定期的な見直し**: 週次・月次・四半期でレビュー
2. **データに基づく判断**: 直感ではなくデータで判断
3. **透明性の確保**: 判断基準と理由を明確に
4. **ステークホルダーとの対話**: 継続的なコミュニケーション
5. **柔軟性**: ビジネス環境の変化に応じて調整

**優先順位付けは、チーム全体の活動です。全員が理解し、協力しましょう！**
