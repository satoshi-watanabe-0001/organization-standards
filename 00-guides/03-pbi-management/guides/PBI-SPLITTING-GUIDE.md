## はじめに

### このガイドの目的

大きすぎるPBIは、以下の問題を引き起こします：
- ⏱️ 見積もりが困難
- 🚫 1スプリントで完了できない
- 📉 進捗が見えにくい
- ⚠️ リスクが高い

このガイドは、**適切なサイズのPBIに分割する実践的な手法**を提供します。

### ガイドの対象者

- **プロダクトオーナー**: Epic/大規模PBIの分割計画
- **開発チーム**: 実装時のPBI分割判断
- **スクラムマスター**: スプリント計画時のPBI調整
- **AIエージェント**: 自動的なPBI分割提案

## なぜPBI分割が必要か

### 大きすぎるPBIの問題

| 問題 | 説明 | 影響 |
|------|------|------|
| **見積もり困難** | 複雑すぎて正確な見積もりができない | スプリント計画が不正確に |
| **完了できない** | 1スプリント（1-2週間）で終わらない | スプリント目標未達成 |
| **進捗不透明** | 部分的な完了が見えない | ステークホルダーへの報告困難 |
| **リスク大** | 問題発見が遅れる | 手戻りが大きい |
| **価値提供遅延** | 全て完了するまで価値を提供できない | フィードバック遅延 |
| **並行作業不可** | 1人しか作業できない | チーム効率低下 |

### 適切なサイズのメリット

| メリット | 説明 |
|---------|------|
| ✅ **見積もり精度向上** | 小さいPBIは見積もりやすい |
| ✅ **早期の価値提供** | 段階的にリリース可能 |
| ✅ **フィードバック早期化** | 早く検証できる |
| ✅ **並行作業可能** | 複数人で分担できる |
| ✅ **リスク軽減** | 問題を早期発見 |
| ✅ **モチベーション向上** | 達成感を得やすい |

## 分割の判断基準

### サイズの目安

| サイズ | ストーリーポイント | 期間 | 判定 | アクション |
|--------|------------------|------|------|----------|
| **Extra Small** | 1-2 | 1-2日 | ✅ 最適 | そのまま実施 |
| **Small** | 3-5 | 3-5日 | ✅ 良好 | そのまま実施 |
| **Medium** | 8 | 1週間 | ⚠️ 要検討 | 分割検討 |
| **Large** | 13 | 2週間 | ❌ 大きすぎ | 分割必須 |
| **Extra Large** | 21+ | 2週間以上 | ❌ Epic化 | Epic化して分割 |

### 分割判断フローチャート

```
PBIの見積もりが完了
    ↓
8ポイント以上？
    ├─ NO → そのまま実施
    └─ YES → 分割を検討
          ↓
    分割可能か？
        ├─ YES → 分割パターンを選択
        │         ↓
        │    分割後、各PBIが8ポイント以下？
        │         ├─ YES → 分割完了
        │         └─ NO → さらに分割
        │
        └─ NO → Epic化
              ↓
         Epic内で複数PBIに分解
```

### 分割が困難なサイン

以下の場合、分割が難しい可能性があります：

- ❌ PBIが曖昧すぎる → まず要件を明確化
- ❌ 技術的不確実性が高い → Spike/POCで調査
- ❌ 依存関係が複雑すぎる → アーキテクチャを見直し

## 8つの分割パターン

### パターン1: ワークフローステップ分割

ユーザーの操作フローを段階的に分割。

#### 適用シーン
- ユーザーストーリーが複数ステップから成る
- 各ステップが独立して価値を提供

#### 実例

**Before（大きすぎるPBI）**:
```
PBI: ユーザー登録機能
見積もり: 21ポイント

含まれる機能:
- 登録フォーム入力
- メール認証
- プロフィール設定
- ウェルカムツアー
```

**After（分割後）**:
```
Epic: ユーザー登録機能

PBI-1: 基本登録機能（5ポイント）
- メールアドレス・パスワードで登録
- 仮登録完了画面表示

PBI-2: メール認証機能（3ポイント）
- 認証メール送信
- メール内リンクで本登録完了

PBI-3: プロフィール設定機能（5ポイント）
- ニックネーム、アイコン設定
- プロフィール保存

PBI-4: ウェルカムツアー（3ポイント）
- 初回ログイン時のガイド表示
```

#### メリット
- ✅ 段階的に価値提供
- ✅ 早期フィードバック
- ✅ MVPから始められる

---

### パターン2: CRUD操作分割

Create、Read、Update、Deleteの各操作で分割。

#### 適用シーン
- データ管理機能
- 各操作が独立している

#### 実例

**Before**:
```
PBI: 商品管理機能
見積もり: 13ポイント

含まれる機能:
- 商品追加
- 商品一覧表示
- 商品編集
- 商品削除
```

**After**:
```
PBI-1: 商品一覧表示（3ポイント）
- 商品リストを表示
- ページネーション

PBI-2: 商品詳細表示（2ポイント）
- 個別商品の詳細表示

PBI-3: 商品追加機能（5ポイント）
- 新規商品登録フォーム
- バリデーション

PBI-4: 商品編集機能（3ポイント）
- 既存商品の編集
```

#### 優先順位の付け方
1. **Read（表示）**: まず見られるようにする
2. **Create（作成）**: 次にデータを増やせるようにする
3. **Update（更新）**: 修正できるようにする
4. **Delete（削除）**: 最後に削除機能を追加

### パターン3: ビジネスルール複雑度分割

シンプルなルールから複雑なルールへ段階的に実装。

#### 適用シーン
- ビジネスロジックが複雑
- 段階的な機能拡張が可能

#### 実例

**Before**:
```
PBI: 配送料計算機能
見積もり: 13ポイント

含まれるルール:
- 基本配送料（地域別）
- 重量による追加料金
- 会員割引
- クーポン適用
- 送料無料キャンペーン
```

**After**:
```
PBI-1: 基本配送料計算（3ポイント）
- 地域別の基本配送料
- 固定料金テーブル

PBI-2: 重量による追加料金（3ポイント）
- 重量しきい値による追加料金

PBI-3: 会員割引適用（2ポイント）
- 会員ランクによる割引率適用

PBI-4: クーポン・キャンペーン適用（5ポイント）
- クーポンコード入力
- 期間限定キャンペーン
```

#### メリット
- ✅ 最小限の機能から開始
- ✅ 複雑さを段階的に追加
- ✅ リスク軽減

### パターン4: データバリエーション分割

データの種類やバリエーションで分割。

#### 適用シーン
- 複数のデータタイプを扱う
- 各タイプの処理が異なる

#### 実例

**Before**:
```
PBI: ファイルアップロード機能
見積もり: 13ポイント

対応ファイル:
- 画像（jpg, png, gif）
- 動画（mp4, mov）
- 文書（pdf, doc）
- その他
```

**After**:
```
PBI-1: 画像アップロード機能（5ポイント）
- jpg, png, gif対応
- サムネイル生成
- 画像圧縮

PBI-2: PDF文書アップロード（3ポイント）
- PDFファイル対応
- プレビュー表示

PBI-3: 動画アップロード（8ポイント）
- mp4, mov対応
- トランスコード処理
→ さらに分割検討
```

### パターン6: パフォーマンス・品質段階分割

基本機能 → 最適化の順で分割。

#### 適用シーン
- まず動くものを作り、後で最適化
- パフォーマンス要件が厳しい

#### 実例

**Before**:
```
PBI: 高速検索機能
見積もり: 13ポイント

要件:
- 基本検索機能
- 全文検索
- パフォーマンス最適化（100ms以内）
- キャッシュ実装
```

**After**:
```
PBI-1: 基本検索機能（5ポイント）
- シンプルなキーワード検索
- 基本的なデータベースクエリ
- 目標: 5秒以内

PBI-2: 検索パフォーマンス最適化（5ポイント）
- インデックス最適化
- クエリチューニング
- 目標: 1秒以内

PBI-3: 検索キャッシュ実装（3ポイント）
- Redis導入
- 人気キーワードのキャッシュ
- 目標: 100ms以内
```

#### メリット
- ✅ 早期に動作確認
- ✅ 測定に基づく最適化
- ✅ 過剰な最適化を回避

### パターン7: インターフェース分割

UI層、API層、データ層で分割。

#### 適用シーン
- フロントエンドとバックエンドで並行作業
- レイヤーごとに担当者が異なる

#### 実例

**Before**:
```
PBI: 商品検索機能（フルスタック）
見積もり: 13ポイント

含まれる作業:
- 検索UI実装
- 検索API実装
- データベース最適化
```

**After**:
```
PBI-1: 検索API実装（5ポイント）
担当: バックエンド
- GET /api/products/search
- クエリパラメータ対応
- モックデータで動作確認

PBI-2: 検索UI実装（5ポイント）
担当: フロントエンド
- 検索フォーム実装
- 検索結果表示
- API連携（PBI-1完了後）

PBI-3: 検索データベース最適化（3ポイント）
担当: バックエンド
- インデックス追加
- クエリ最適化
```

#### メリット
- ✅ 並行作業可能
- ✅ 各レイヤーの専門性活用
- ✅ API仕様を先に確定

### パターン8: スパイク分離

技術調査を独立したSpike/POCとして分割。

#### 適用シーン
- 技術的不確実性が高い
- 見積もりが困難

#### 実例

**Before**:
```
PBI: リアルタイム通知機能
見積もり: 21ポイント（不確実性大）

不明点:
- WebSocketとSSEどちらを使うか不明
- スケーラビリティ不明
- 既存システムとの統合方法不明
```

**After**:
```
PBI-1: [SPIKE] リアルタイム通知技術検証（3ポイント）
タイムボックス: 2日
- WebSocket vs SSE比較
- スケーラビリティ検証
- 技術選定レポート作成

PBI-2: リアルタイム通知基盤実装（5ポイント）
前提: PBI-1で選定した技術を使用
- 通知サーバー実装
- 接続管理機能

PBI-3: 通知UI実装（5ポイント）
- 通知表示コンポーネント
- リアルタイム更新

PBI-4: 通知機能統合（3ポイント）
- 既存システムとの統合
- E2Eテスト
```

#### メリット
- ✅ リスク早期発見
- ✅ 見積もり精度向上
- ✅ 技術選定の根拠明確化

## PBIタイプ別の分割戦略

### NPI（新規プロジェクト立ち上げ）の分割

**特徴**: 非常に大きい（通常21ポイント以上）

**推奨分割方法**:
```
Epic: 新規プロジェクト立ち上げ

Phase別に分割:
├─ PBI-1: Phase 1（プロジェクト構造）（8ポイント）
├─ PBI-2: Phase 2（設計）（5ポイント）
├─ PBI-3: Phase 3（基本実装）（13ポイント）← さらに分割
│   ├─ PBI-3-1: バックエンド基盤（5ポイント）
│   ├─ PBI-3-2: フロントエンド基盤（5ポイント）
│   └─ PBI-3-3: データベース構築（3ポイント）
├─ PBI-4: Phase 4（テスト）（5ポイント）
└─ PBI-5: Phase 5（デプロイ準備）（3ポイント）
```

---

### NFD（新規機能開発）の分割

**推奨パターン**: ワークフローステップ分割、CRUD分割

**例**:
```
Epic: お気に入り機能

├─ PBI-1: お気に入り追加機能（3ポイント）
├─ PBI-2: お気に入り一覧表示（3ポイント）
└─ PBI-3: お気に入り削除機能（2ポイント）
```

---

### ENH（既存機能改修）の分割

**推奨パターン**: パフォーマンス段階分割、ビジネスルール分割

**例**:
```
Epic: 検索パフォーマンス改善

├─ PBI-1: インデックス最適化（3ポイント）
├─ PBI-2: クエリチューニング（3ポイント）
└─ PBI-3: キャッシュ層追加（5ポイント）
```

---

### ARC（アーキテクチャ改善）の分割

**推奨パターン**: サービス分割、段階的移行

**例**:
```
Epic: マイクロサービス化

├─ PBI-1: ユーザーサービス分離（13ポイント）← さらに分割
│   ├─ PBI-1-1: サービス基盤構築（5ポイント）
│   ├─ PBI-1-2: データベース分離（5ポイント）
│   └─ PBI-1-3: API Gateway統合（3ポイント）
├─ PBI-2: 商品サービス分離（13ポイント）
└─ PBI-3: 注文サービス分離（13ポイント）
```

## 分割時の注意点

### 注意点1: 価値を提供し続ける

各PBIが独立して価値を提供できるように分割。

#### ❌ Bad Example
```
PBI-1: データベーステーブル作成（3ポイント）
PBI-2: バックエンドAPI実装（5ポイント）
PBI-3: フロントエンドUI実装（5ポイント）

問題: PBI-1だけでは価値なし
```

#### ✅ Good Example
```
PBI-1: 商品一覧表示機能（5ポイント）
- データベース設計
- API実装
- UI実装
→ ユーザーが商品を見られる（価値あり）

PBI-2: 商品詳細表示機能（3ポイント）
→ より詳しい情報が見られる（追加価値）
```

---

### 注意点2: 依存関係を最小化

PBIが独立して実装できるように分割。

#### 依存関係の扱い方

**許容できる依存**:
- 技術基盤（データベース、認証等）
- 共通ライブラリ

**避けるべき依存**:
- 特定PBIの完了待ち
- 順次実行しかできない設計

---

### 注意点3: テスト可能性を保つ

各PBIが独立してテストできるように分割。

**チェックポイント**:
- ✅ 単体テスト可能か？
- ✅ 統合テスト可能か？
- ✅ E2Eテストのシナリオが作れるか？

---

### 注意点4: ドキュメントの一貫性

分割後もドキュメントの一貫性を保つ。

**推奨**:
- Epic配下に全PBIをまとめる
- 共通の前提条件をEpicに記載
- 各PBIから Epicへのリンクを設定

- ✅ 各PBI が8ポイント以下

## 分割後の管理

### Epic管理

分割後のPBIは、Epicで管理します。

#### Epic構造
```
Epic: [機能名]
├─ 概要・目的
├─ ビジネス価値
├─ 全体スコープ
├─ タイムライン
└─ PBI一覧
    ├─ PBI-1: [MVP機能]（優先度: High）
    ├─ PBI-2: [追加機能]（優先度: Medium）
    └─ PBI-3: [拡張機能]（優先度: Low）
```

### 優先順位の付け方

#### MoSCoW法

| 分類 | 説明 | 実装タイミング |
|------|------|--------------|
| **Must Have** | 必須機能（MVP） | 最初のスプリント |
| **Should Have** | 重要だが必須ではない | 2-3スプリント目 |
| **Could Have** | あると良い | 余裕があれば |
| **Won't Have** | 今回は実装しない | 将来検討 |

#### 優先順位の判断基準

1. **ビジネス価値**: ROIが高いものを優先
2. **リスク**: リスクが高いものを早期に
3. **依存関係**: 他のPBIの前提となるものを優先
4. **学習**: 不確実性が高いものを早期に検証

## AIエージェント向けガイダンス

### AI自動分割フロー

AIエージェントは、以下のフローで PBIを自動分割してください：

```python
def auto_split_pbi(pbi):
    # Step 1: サイズチェック
    if pbi.story_points <= 8:
        return [pbi]  # 分割不要
    
    # Step 2: 分割パターン選択
    pattern = select_split_pattern(pbi)
    
    # Step 3: 分割実行
    split_pbis = apply_pattern(pbi, pattern)
    
    # Step 4: 各分割PBIの検証
    for split_pbi in split_pbis:
        if not validate_split_pbi(split_pbi):
            # 再分割または調整
            split_pbi = adjust_pbi(split_pbi)
    
    # Step 5: 依存関係設定
    set_dependencies(split_pbis)
    
    return split_pbis

def select_split_pattern(pbi):
    # PBIの特性から最適なパターンを選択
    if has_workflow_steps(pbi):
        return "ワークフローステップ分割"
    elif is_crud_operation(pbi):
        return "CRUD操作分割"
    elif has_complex_rules(pbi):
        return "ビジネスルール分割"
    # ... 他のパターン判定
```

## まとめ

### 分割の原則

1. **8ポイント以下**: 各PBIは8ポイント以下を目指す
2. **価値提供**: 各PBIが独立して価値を提供
3. **最小化依存**: 依存関係を最小化
4. **テスト可能**: 各PBIが独立してテスト可能
5. **段階的**: MVPから始めて段階的に拡張

### 8つの分割パターン

1. ✅ ワークフローステップ分割
2. ✅ CRUD操作分割
3. ✅ ビジネスルール複雑度分割
4. ✅ データバリエーション分割
5. ✅ プラットフォーム/デバイス分割
6. ✅ パフォーマンス・品質段階分割
7. ✅ インターフェース分割
8. ✅ スパイク分離

### 次のステップ

1. **大きすぎるPBIを特定**: 現在のバックログをレビュー
2. **分割パターン選択**: 適切なパターンを選択
3. **Epic作成**: 分割後のPBIをEpicで管理
4. **優先順位付け**: MoSCoW法で優先順位を決定
5. **スプリント計画**: 優先度の高いPBIから着手
