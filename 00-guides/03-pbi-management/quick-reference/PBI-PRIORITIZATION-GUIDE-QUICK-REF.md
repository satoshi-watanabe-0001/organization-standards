## 🎯 優先順位付けの4大原則

### 1. ビジネス価値の評価

**収益インパクトスコア**:
```
収益インパクトスコア = 予想収益増加額（年間） ÷ 100万円
```

**ユーザー満足度レベル**:
- **Critical (10)**: ユーザーが製品を使えない致命的な課題
- **High (7-9)**: ユーザーの主要業務に大きく影響
- **Medium (4-6)**: ユーザー体験を改善、業務効率化
- **Low (1-3)**: あれば便利だが必須ではない

### 2. 技術的複雑度の評価

**実装難易度（Tシャツサイズ）**:
| サイズ | 想定工数 | スコア |
|--------|----------|--------|
| **XS** | 1-2日 | 1 |
| **S** | 3-5日 | 2 |
| **M** | 1-2週 | 3 |
| **L** | 2-4週 | 5 |
| **XL** | 1-2ヶ月 | 8 |

### 3. リスク評価

**リスクスコア計算**:
```
リスクスコア = 発生確率（%） × 影響度（1-10）
技術リスク優先度 = Severity（1-10） × Probability（0.1-1.0）
```

**時間的緊急性**:
- **今すぐ**: 24時間以内（HOT）
- **今週中**: 1週間以内
- **今月中**: 1ヶ月以内
- **今四半期中**: 3ヶ月以内
- **いつでも**: 特定の期限なし

### 4. ROI（投資対効果）分析

**ROI計算式**:
```
ROI = (期待リターン - 投資コスト) ÷ 投資コスト × 100%
```

**優先度判定**:
| ROI範囲 | 優先度 | 判断 |
|---------|--------|------|
| **500%以上** | 最高 | 最優先で実施 |
| **200-500%** | 高 | 積極的に実施 |
| **100-200%** | 中 | 条件次第で実施 |
| **50-100%** | 低 | 慎重に検討 |
| **50%未満** | 最低 | 実施を見送り |

## 🔢 優先順位付け手法（5つ）

### 1. MoSCoW法 ★★★★★

**カテゴリ分類**:
- **Must have**: 必須、これがないとリリース不可
- **Should have**: 重要、可能な限り含めるべき
- **Could have**: あれば良い、余裕があれば実装
- **Won't have**: 今回は実装しない

**適用場面**: 初期段階、シンプルな分類が必要な時

### 2. RICE評価 ★★★★☆

**計算式**:
```
RICEスコア = (Reach × Impact × Confidence) ÷ Effort

Reach: 影響を受けるユーザー数（月間）
Impact: ユーザーへの影響度（0.25/0.5/1/2/3）
Confidence: 確信度（%、例: 80% = 0.8）
Effort: 必要な工数（人月）
```

**適用場面**: 定量的評価が必要な時

### 3. WSJF（Weighted Shortest Job First） ★★★★☆

**計算式**:
```
WSJF = CoD（Cost of Delay） ÷ Job Size

CoD = ビジネス価値 + 時間的価値 + リスク軽減・機会創出
```

**適用場面**: SAFe/アジャイル環境、継続的な優先順位調整

### 4. Kano Model ★★★☆☆

**3つのカテゴリ**:
- **基本品質**: なければ不満、あっても当然（例：ログイン機能）
- **性能品質**: あればあるほど満足度向上（例：検索速度）
- **魅力品質**: あれば驚きと喜び（例：AI推奨機能）

**適用場面**: ユーザー満足度重視の製品企画

### 5. Value vs Effort Matrix ★★★★☆

**4象限マトリックス**:
```
        高価値
    Quick Wins | Major Projects
    -----------|---------------
    Fill-ins   | Time Sinks
        低価値
        低工数        高工数
```

- **Quick Wins**: 優先度最高
- **Major Projects**: 慎重に計画
- **Fill-ins**: 空き時間に実施
- **Time Sinks**: 実施を避ける

## 📊 PBIタイプ別優先順位戦略

| タイプ | 略称 | 基本優先度 | 重点評価項目 |
|--------|------|-----------|------------|
| **緊急修正** | HOT | 最高 | 影響範囲、ビジネスインパクト |
| **バグ修正** | BUG | 高〜中 | Severity、影響ユーザー数 |
| **新規プロジェクト** | NPI | 高 | 戦略的重要性、市場機会 |
| **新規機能** | NFD | 中〜高 | ビジネス価値、ユーザー需要 |
| **既存機能改修** | ENH | 中 | ROI、ユーザーフィードバック |
| **アーキテクチャ改善** | ARC | 中 | 長期的価値、技術的負債削減 |
| **リファクタリング** | REF | 低〜中 | 開発効率改善、保守性向上 |
| **実験的機能** | POC | 低 | 学習価値、リスク検証 |

### HOT（緊急修正）の優先順位判定

**重大度（Severity）**:
- **Critical**: システムダウン、データ損失 → 即座対応
- **High**: 主要機能停止、多数ユーザー影響 → 24時間以内
- **Medium**: 一部機能不具合 → 通常対応

**影響範囲**:
```
優先度スコア = 影響ユーザー数 × Severity（1-10）
```

### BUG（バグ修正）の優先順位判定

**評価基準**:
| Severity | 影響範囲 | 対応期限 | 優先度 |
|----------|---------|---------|--------|
| **Critical** | 全ユーザー | 24時間以内 | P0 |
| **High** | 多数ユーザー | 3日以内 | P1 |
| **Medium** | 一部ユーザー | 1週間以内 | P2 |
| **Low** | 限定的 | 次回リリース | P3 |

## 🤖 AIエージェント向けクイックガイド

### 自動優先順位提案の手順

1. **PBIタイプの確認** → タイプ別戦略を適用
2. **定量評価の実施** → RICE/WSJF/ROI計算
3. **リスク評価** → Severity × Probability
4. **総合スコアの算出** → 重み付け平均
5. **推奨優先度の提示** → P0/P1/P2/P3/P4
6. **根拠の明示** → 各評価項目のスコア表示

### 定量評価テンプレート

```json
{
  "pbi_id": "PBI-XXX",
  "pbi_type": "NFD",
  "scores": {
    "business_value": 8,
    "user_satisfaction": 7,
    "revenue_impact": 5,
    "strategic_importance": 6,
    "technical_complexity": 3,
    "technical_risk": 2,
    "time_risk": 4,
    "roi_percentage": 250
  },
  "rice_score": 24.0,
  "wsjf_score": 7.5,
  "recommended_priority": "P1",
  "rationale": "高いビジネス価値（8）とROI（250%）、低い技術リスク（2）により、P1を推奨"
}
```

### 判断基準の重み付け

**優先度決定の重み係数**:
```
総合スコア = 
  ビジネス価値 × 0.35 +
  ユーザー満足度 × 0.25 +
  技術的複雑度 × 0.15（逆数） +
  リスク × 0.15 +
  ROI × 0.10
```

## 🚨 よくある失敗パターン（Top 5）

### 1. 声の大きいステークホルダーに流される

**問題**: 客観的評価より政治的圧力を優先
**対策**: 定量評価を用いた透明な意思決定

### 2. 技術的負債を無視する

**問題**: 短期的な機能開発ばかり優先し、リファクタリングを後回し
**対策**: 20%ルール（全工数の20%を技術的改善に充てる）

### 3. すべてを「優先度:高」にする

**問題**: 優先順位付けの意味がなくなる
**対策**: 相対評価、MoSCoW法で強制分類

### 4. 依存関係を考慮しない

**問題**: 依存先が未完了で着手できない
**対策**: JIRA Linkageで依存関係を可視化

### 5. 優先順位を固定化する

**問題**: 状況変化に対応できない
**対策**: スプリントごとに見直し、柔軟に調整
