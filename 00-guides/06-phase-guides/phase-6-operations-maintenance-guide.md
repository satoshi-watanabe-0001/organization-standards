# Phase 6: 運用・保守ガイド

> デプロイから継続的改善へ - 軽量ナビゲーションガイド

**対象**: Devin、Cursor、その他の自律型AIエージェント  
**前提**: Phase 5（デプロイメント）が完了し、アプリケーションが本番環境で稼働している  
**目的**: システムの安定稼働を維持し、継続的な改善を実施する

- **重複回避**: 詳細は既存ドキュメントを参照

## ⚠️ 禁止事項チェック

### このフェーズ特有の禁止事項
- **直接本番修正**: 本番環境で直接コードを修正してはいけない（必ずPR経由）
- **モニタリング無視**: アラートやメトリクスを確認せずに運用してはいけない
- **ドキュメント更新放置**: 変更後にドキュメントを更新しないで放置してはいけない
- **ログ削除**: トラブルシューティングに必要なログを削除してはいけない

### 全フェーズ共通禁止事項
- **CI/CD設定の無断変更**: 明示的な指示がない限り、GitHub Actions、CircleCI、Jenkins等のCI/CD設定ファイルを変更してはいけない
- **本番環境への直接変更**: 本番データベース、本番サーバー、本番設定ファイルを直接変更してはいけない
- **セキュリティ設定の独断変更**: 認証・認可・暗号化等のセキュリティ設定を独自判断で変更してはいけない
- **組織標準外技術の無断導入**: プロジェクトで使用していない新しいライブラリ・フレームワーク・言語を独断で導入してはいけない
- **プロジェクト構造の大幅変更**: ディレクトリ構造、命名規則、アーキテクチャパターンを独断で大きく変更してはいけない

### 📚 詳細確認
禁止事項の詳細、具体例、例外ケースについては以下のドキュメントを参照してください：
- [AI開発タスクの禁止事項](../../10-governance/ai-task-prohibitions.md)
- [禁止事項チェックリスト](../../10-governance/ai-task-prohibitions-checklist.md)

## 🔄 Phase 6 のワークフロー: 6ステップ

### **Step 6.1: モニタリング設定（初期設定）** (30-45分)

**実行内容**:
1. メトリクス設定
2. アラート設定
3. ダッシュボード構築

**チェックリスト**:
- [ ] 主要メトリクスを設定（CPU、メモリ、ディスク、ネットワーク）
- [ ] アプリケーションメトリクスを設定（レスポンスタイム、エラー率等）
- [ ] アラート閾値を設定
- [ ] アラート通知先を設定（Slack、メール等）
- [ ] ダッシュボードを構築
- [ ] ヘルスチェックを設定

**🔴 必須参照**:
- [`05-technology-stack/monitoring-logging.md`](../../05-technology-stack/monitoring-logging.md)
  - **このドキュメントの記載内容に従ってモニタリングを設定**
  - **メトリクス、アラート、ダッシュボードの設定方法を確認**

**設定すべき主要メトリクス**:
- **インフラメトリクス**: CPU使用率、メモリ使用率、ディスク使用率、ネットワーク帯域
- **アプリケーションメトリクス**: リクエスト数、エラー率、レスポンスタイム
- **ビジネスメトリクス**: アクティブユーザー数、トランザクション数

**アラート設定例（Prometheus + Grafana）**:
```yaml
groups:
  - name: application_alerts
    rules:
      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.05
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "High error rate detected"
          
      - alert: HighResponseTime
        expr: histogram_quantile(0.95, http_request_duration_seconds) > 1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High response time detected"
```

### **Step 6.2: ログ管理設定（初期設定）** (20-30分)

**実行内容**:
1. ログ収集設定
2. ログ分析ルール設定
3. ログ保持ポリシー設定

**チェックリスト**:
- [ ] アプリケーションログを集中管理システムに送信
- [ ] エラーログのフィルタリングルールを設定
- [ ] ログレベル（DEBUG、INFO、WARN、ERROR）を適切に設定
- [ ] ログ保持期間を設定
- [ ] ログ分析ダッシュボードを構築

**🔴 必須参照**:
- [`05-technology-stack/monitoring-logging.md`](../../05-technology-stack/monitoring-logging.md)
  - **ログ管理の設定方法を確認**

**ログ収集設定例（Fluentd）**:
```yaml
<source>
  @type tail
  path /var/log/app/*.log
  pos_file /var/log/fluentd/app.pos
  tag app.logs
  <parse>
    @type json
  </parse>
</source>

<match app.logs>
  @type elasticsearch
  host elasticsearch.example.com
  port 9200
  index_name app-logs
</match>
```

### **Step 6.4: インシデント対応（発生時）** (可変: 15分-数時間)

**実行内容**:
1. インシデントの検知
2. 初動対応と影響範囲の特定
3. 復旧作業
4. 事後分析（ポストモーテム）

**チェックリスト**:
- [ ] インシデントの重大度を評価（Critical / High / Medium / Low）
- [ ] 影響範囲を特定（ユーザー数、機能等）
- [ ] 初動対応を実施（ロールバック、サービス停止等）
- [ ] 根本原因を調査
- [ ] 恒久対策を実施
- [ ] インシデントレポートを作成
- [ ] ポストモーテムを実施

**🔴 必須参照**:
- [`03-development-process/incident-management.md`](../../03-development-process/incident-management.md)
  - **このドキュメントの記載内容に従ってインシデント対応を実施**
  - **インシデント対応フロー、エスカレーション、ポストモーテムの方法を確認**

**インシデント対応フロー**:
```
1. 検知 → 2. 評価 → 3. 初動対応 → 4. 復旧 → 5. 事後分析 → 6. 再発防止
```

**ポストモーテムテンプレート**:
```markdown
# インシデントレポート: [タイトル]

## 概要
- **発生日時**: 2025-11-05 10:00
- **復旧日時**: 2025-11-05 10:45
- **影響範囲**: 全ユーザー
- **重大度**: Critical

## 何が起きたか
APIサーバーがメモリ不足でクラッシュ

## 根本原因
メモリリークが原因

## 対応内容
1. サーバーを再起動（10:15）
2. メモリリークの原因を特定（10:30）
3. 修正コードをデプロイ（10:45）

## 再発防止策
- メモリ使用量のアラートを追加
- コードレビューでメモリリークチェックを強化
```

## 🔍 実践的な活用方法

### 基本的なワークフロー

1. **初期設定（デプロイ直後）**
   - 必須: モニタリング設定、ログ管理設定

2. **日常運用**
   - 必須: 日常モニタリング
   - 発生時: インシデント対応

3. **定期的な活動**
   - 必須: 変更管理、技術的負債管理
   - 継続的: ドキュメント更新

### 運用タイプ別のアプローチ

**新規システム**:
- **初期設定に時間をかける** (2-3時間)
- モニタリングとアラートを詳細に設定
- 最初の1-2週間は監視を強化

**既存システム**:
- **既存設定の見直し** (1-2時間)
- 不要なアラートの削減
- 新機能のモニタリング追加

**インシデント多発時**:
- **根本原因の分析を優先**
- 技術的負債の解消計画を策定
- モニタリングとアラートを強化

## ⚠️ 重要な原則

### ドキュメント参照の原則
- **参照ドキュメントの記載内容が最優先**
- 本ガイドは参照先への案内役
- 疑問点は該当ドキュメントを直接確認

### 運用の原則
- **予防が最優先**: モニタリングとアラートで問題を早期発見
- **迅速な対応**: インシデントは迅速に対応
- **継続的改善**: ポストモーテムで再発防止
- **ドキュメント更新**: 変更はすぐにドキュメント化
- **技術的負債管理**: 定期的に対応

### Phase 6の特徴
- **終わりがない**: 継続的な運用活動
- **Phase 0に戻る**: 改善のため新しいPBIを作成してサイクルを回す
- **フィードバックループ**: 運用で得た知見を次の開発に活かす

## まとめ

Phase 6は**システムの安定稼働と継続的改善を担う重要なフェーズ**です。

- **目的**: システムの安定稼働を維持し、継続的な改善を実施する
- **成果物**: モニタリングダッシュボード、インシデントレポート、変更管理記録、技術的負債リスト、更新ドキュメント
- **所要時間**: 初期設定で2-3時間、日常運用は継続的

**最も重要な原則**: 
- モニタリングとアラートで問題を早期発見してください
- インシデントには迅速に対応してください
- ポストモーテムで根本原因を分析し、再発防止策を講じてください
- 技術的負債を定期的に管理し、対応してください
- 変更は必ずドキュメント化してください

**Phase 0に戻る**: 運用で得た改善点や新機能の要望をPBIとして記録し、Phase 0から新しいサイクルを開始してください。

疑問や矛盾がある場合は、より具体的で新しいドキュメントを優先してください。
