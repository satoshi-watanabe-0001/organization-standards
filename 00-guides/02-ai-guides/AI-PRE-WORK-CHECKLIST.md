# AI作業開始前チェックリスト

> AIエージェントが作業を開始する前に確認すべき事項の統合チェックリスト

**対象**: Devin、Cursor、その他の自律型AIエージェント  
**目的**: 作業開始前の確認漏れを防止し、効率的な開発を実現  
**使用タイミング**: 初回起動時、PBI受領時、各フェーズ開始時

---

## 📋 このチェックリストの使い方

### 3つのチェックリスト

このドキュメントには3種類のチェックリストがあります：

1. **初回起動時チェックリスト** - 最初の1回のみ実施（30-60分）
2. **PBI受領時チェックリスト** - PBIを受け取るたびに実施（5-10分）
3. **フェーズ開始時チェックリスト** - 各フェーズ開始時に実施（2-5分）

### 重要度の凡例

- 🔴 **必須** - スキップ不可、確認必須
- 🟡 **推奨** - 確認推奨、スキップ時は記録
- 🟢 **任意** - 必要に応じて確認

- [ ] 🔴 **フェーズスキップ判断フローを実施した**

  **Q2: プロジェクト構造・技術スタック確定済みか？**
  - [ ] Yes → Phase 1 スキップ
  - [ ] No → Phase 1 実施

  **Q3: 設計変更が必要か？**
  - [ ] Yes → Phase 2 実施

  **Q5: デプロイが必要か？**

  - [ ] No → Phase 5 スキップ

### 人間への確認が必要か判断（エスカレーション判断）

**🆕 v1.3.0 拡張**: エスカレーション判断を実行可能なフローに改善

#### 3段階の判断レベル

- 🔴 **即座にエスカレーション**: 作業を停止して人間に確認
- 🟡 **実装後に確認**: 標準的な想定で実装し、完了後に人間レビュー要求
- ✅ **自律判断可能**: AIが自律的に判断して進行

#### PBI受領時の判断（Phase 0開始前）

##### PBI内容の明確さレベル評価

- [ ] 🔴 **Level 3（非常に不明確）→ 即座にエスカレーション**
  - 受入基準が曖昧（「高速化」「改善」等の定性的表現のみ）
  - 複数の解釈が可能で、どれが正しいか判断できない
  - 技術的制約が不明で影響範囲が判断できない
  
- [ ] 🟡 **Level 2（やや不明確）→ 標準的な想定で進行**
  - 一部の詳細は不明だが、標準的なアプローチで進められる
  - 不明点があるが、実装しながら明確化できる範囲
  
- [ ] ✅ **Level 1（明確）→ 自律実行可能**
  - 受入基準が定量的（「レスポンス時間1秒以内」等）
  - 技術的制約が明示されている

##### エスカレーション実行（Level 3の場合）

エスカレーション記録テンプレート:
```markdown
## 🚨 エスカレーション: PBI-XXX

### 不明点・懸念事項
1. **受入基準の解釈が不明確**
   - 問題: [具体的な問題]
   - 想定される解釈: A案/B案/C案
   - 推奨: [推奨案]

### 暫定的な想定（人間確認待ち）
- [想定1]
- [想定2]

### 人間の判断（回答欄）
- [ ] A案を採用
- [ ] B案を採用
```

#### 実装中の判断（Phase 3）

##### 想定外の技術的制約を発見した場合

**判断フロー**:
```
技術的制約を発見
    ↓
代替案が組織標準内?
    ├─ Yes → 🟡 実装後に確認（PR説明文に記載）
    └─ No  → 🔴 即座にエスカレーション
```

##### 設計変更の必要性を発見した場合

- 🔴 **即座にエスカレーション**: 影響範囲が広範囲（3+ファイル、他モジュールに影響）
- 🟡 **実装後に確認**: 影響範囲が限定的（1-2ファイル、リファクタリングレベル）

#### 自律的に判断可能な項目（エスカレーション不要）

- ✅ 実装の詳細（変数名、関数名、ファイル分割、ロジック実装）
- ✅ テストの作成（ユニットテスト、テストデータ、モック/スタブ）
- ✅ ドキュメント作成（README, コメント、API仕様書）
- ✅ 軽微なリファクタリング（可読性向上、重複削減、メソッド抽出）

  - [ ] 実行すべきフェーズを決定した

## 🔍 AI自己レビューワークフロー（v1.3.0新規追加）

> **重要**: AIが実装中および実装後に品質を確保するための段階的レビューワークフロー

**目的**: 問題の早期発見、品質保証、人間レビュー時間の短縮  
**適用フェーズ**: 主にPhase 3（実装）、Phase 4（レビュー・QA）  
**期待効果**: レビュー往復-50%、人間レビューコメント-67%、総開発時間-5%

### 3段階レビューアプローチ

#### Stage 1: プレ実装レビュー（Before Implementation）
**タイミング**: コーディング開始前（5-10分）

- [ ] 🔴 設計との整合性確認（設計書、アーキテクチャパターン、既存コードとの一貫性）
- [ ] 🔴 実装計画のレビュー（実装順序、テスト戦略、リスク特定）
- [ ] 🟡 代替案の検討（トレードオフ、パフォーマンス、保守性）

#### Stage 2: インクリメンタルレビュー（During Implementation）
**タイミング**: 各機能単位の実装完了時（5-15分/機能）

**自動チェック**:
- [ ] 🔴 Linter実行（エラー0必須）
- [ ] 🔴 フォーマッター実行
- [ ] 🔴 型チェック実行（該当する場合）
- [ ] 🔴 ユニットテスト実行（カバレッジ≥80%）

**手動レビュー**:
- [ ] 🟡 命名の適切性
- [ ] 🟡 関数設計（単一責任原則、50行以内推奨）
- [ ] 🟡 エラーハンドリング
- [ ] 🟡 コメントの品質

#### Stage 3: ポスト実装レビュー（After Implementation）
**タイミング**: フェーズ完了時、PR作成前（30-60分）

**包括的自動チェック**:
- [ ] 🔴 全テストスイート実行（Unit, Integration, E2E）
- [ ] 🔴 静的解析ツール実行（SonarQube等、スコアA or B）
- [ ] 🔴 セキュリティスキャン実行（Critical 0, High 0必須）
- [ ] 🔴 コード品質メトリクス確認（複雑度≤10、重複≤5%）

**統合的品質確認**:
- [ ] 🔴 受入基準の確認（全て満たしているか）
- [ ] 🔴 動作確認（ローカル環境、回帰テスト）
- [ ] 🔴 ドキュメント完全性（README, API Doc, CHANGELOG）

**人間レビュー準備**:
- [ ] 🔴 PR説明文の完成（What, Why, How, 影響範囲、テスト方法）
- [ ] 🟡 レビュー観点の明示

### レビュー基準クイックリファレンス

| 観点 | ツール | 基準 | 自動化 |
|------|--------|------|--------|
| カバレッジ | Jest, pytest-cov | ≥80% | ✅ |
| 複雑度 | ESLint, Radon | ≤10 | ✅ |
| Linterエラー | ESLint, Pylint | 0 | ✅ |
| 脆弱性 | npm audit | Critical 0, High 0 | ✅ |

## 💡 よくある質問（FAQ）

### Q1: 全てのチェックリストを毎回実施する必要がありますか？

**A**: いいえ。
- **初回起動チェックリスト** → 最初の1回のみ
- **PBI受領時チェックリスト** → PBIを受け取るたびに
- **フェーズ開始時チェックリスト** → 実施する各フェーズで

### Q2: チェックリストの項目をスキップできますか？

**A**: 重要度に応じて：
- 🔴 **必須** → スキップ不可
- 🟡 **推奨** → スキップ可能だが記録を推奨
- 🟢 **任意** → 必要に応じて実施

### Q3: 緊急修正（Hotfix）の場合はどうすればいいですか？

**A**: 簡易版で実施：
- PBI受領時チェックリスト → 5分で必須項目のみ
- Phase 0 → 簡易版（15-30分）
- Phase 3, 4 → 必須
- 後で詳細レビュー・ドキュメント補完を実施

### Q4: チェックリストの記録は必須ですか？

**A**: 推奨します：
- スキップしたフェーズの理由を明記
- 後から判断根拠を確認できる
- チーム内で一貫性を保つため
