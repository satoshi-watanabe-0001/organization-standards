## はじめに

### このプロセスの目的

AIが自律的に作業を進める中で、以下のような課題を発見した場合に体系的に記録します:

✅ **ドキュメントの不備・矛盾**: 改善すべきドキュメント箇所の特定  
✅ **プロセスの非効率性**: ボトルネックや無駄な手順の発見  
✅ **技術的負債**: 将来的に改善すべきコード・設計の記録  
✅ **ツール・インフラの問題**: 開発環境・CI/CDの問題点  
✅ **標準・ルールのギャップ**: 実態と標準のズレの指摘

### 基本原則

```yaml
基本原則:
  proactive_recording: "問題を発見したら即座に記録"
  structured_format: "統一されたテンプレートで記録"
  actionable: "具体的な改善案を含める"
  prioritized: "優先順位を明確にする"
  tracked: "定期的にレビュー・対応"
```

## 課題の種類

### カテゴリ1: ドキュメント関連 📄

```yaml
ドキュメントの不備:
  - 情報の欠落（必要な情報が記載されていない）
  - 情報の古さ（最新の状態を反映していない）
  - 矛盾（複数ドキュメント間で指示が異なる）
  - 曖昧さ（解釈が複数可能）
  - 例の不足（具体例がなく理解困難）
  - アクセス性（必要なドキュメントが見つからない）

🚨 重要: 組織標準ドキュメントの直接修正は絶対禁止
  - AIがドキュメントを直接修正してはいけない
  - タイポ・誤字・リンク切れの修正も禁止
  - 軽微な改善も禁止
  - 必ず課題起票プロセスを経る
  - 人間がレビュー・承認・修正を実施
  - 詳細: AI-DOCUMENT-MODIFICATION-POLICY.md

記録すべき情報:
  - ドキュメント名・パス
  - 問題箇所（セクション番号、行番号）
  - 具体的な問題内容
  - 改善提案（diffスタイルで記述）
  - 影響度（作業への影響）
  - 修正は人間が実施（AIは修正しない）
```

---

### カテゴリ2: プロセス関連 ⚙️

```yaml
プロセスの非効率性:
  - 冗長な手順（統合可能な複数ステップ）
  - ボトルネック（待ち時間が長い箇所）
  - 不要な承認（自動化可能な人間承認）
  - 重複作業（複数箇所で同じ作業）
  - 手動作業（自動化可能な手動タスク）

記録すべき情報:
  - プロセス名・Phase
  - 具体的な非効率箇所
  - 現状の所要時間
  - 改善提案
  - 期待される効果（時間削減、品質向上等）
```

---

### カテゴリ3: 技術的負債 🔧

```yaml
技術的負債:
  - レガシーコード（保守困難なコード）
  - テスト不足（カバレッジ低い箇所）
  - 設計の問題（拡張性・保守性の欠如）
  - パフォーマンス問題（最適化の余地）
  - セキュリティ懸念（潜在的な脆弱性）
  - 依存関係の問題（古いライブラリ等）

記録すべき情報:
  - 該当コード・モジュールの場所
  - 具体的な問題内容
  - ビジネスリスク・技術リスク
  - 改善提案
  - 推奨される対応時期
```

---

### カテゴリ4: ツール・インフラ関連 🛠️

```yaml
ツール・インフラの問題:
  - CI/CDの不安定性（フレイキーテスト、タイムアウト）
  - 開発環境の問題（セットアップ困難、動作不安定）
  - ツールの不足（必要なツールが欠けている）
  - パフォーマンス問題（ビルド・テストが遅い）
  - 設定の問題（不適切な設定、ドキュメント不足）

記録すべき情報:
  - ツール・インフラの名称
  - 具体的な問題内容
  - 発生頻度
  - 作業への影響
  - 改善提案
```

---

### カテゴリ5: 標準・ルールのギャップ 📏

```yaml
標準と実態のズレ:
  - 標準が実態に合っていない
  - 標準が守られていない（既存コードが標準違反）
  - 標準が厳しすぎる（実現困難）
  - 標準が緩すぎる（品質問題）
  - 標準の解釈が分かれる（曖昧な記述）

記録すべき情報:
  - 該当する標準・ルール
  - 具体的なギャップ内容
  - 実態の状況
  - 改善提案（標準修正 or 実態改善）
  - 影響範囲
```

## 課題記録のタイミング

### 即座に記録すべきケース 🚨

```yaml
即座に記録:
  - 作業が完全にブロックされる問題
  - セキュリティに関わる問題
  - データ損失リスクがある問題
  - 複数PBIに共通する問題（影響大）
  
対応:
  1. 課題を即座に記録
  2. 人間にエスカレーション
  3. ワークアラウンドを探す
```

---

### Phase完了時に記録すべきケース ⚠️

```yaml
Phase完了時に記録:
  - 作業を進める上で気づいた非効率性
  - ドキュメントの軽微な不備
  - 将来的に改善すべき技術的負債
  - ツールの軽微な問題
  
対応:
  1. 作業を継続
  2. Phase完了時に課題を記録
  3. 週次レビューで報告
```

---

### PBI完了時にまとめて記録すべきケース 💡

```yaml
PBI完了時に記録:
  - 改善アイデア（必須ではない）
  - プロセスの最適化案
  - ドキュメント改善提案
  - コードの軽微なリファクタリング案
  
対応:
  1. PBI完了まで作業を継続
  2. PBI完了時に課題をまとめて記録
  3. 月次レビューで報告
```

## 課題の優先順位付け

### 優先度の判定基準

```yaml
Critical（クリティカル）- 即座に対応必須:
  条件:
    - 作業が完全にブロックされる
    - セキュリティリスクがある
    - データ損失リスクがある
    - 本番環境に影響する
  
  対応:
    - 即座に人間にエスカレーション
    - 24時間以内に対応開始

High（高）- 早急に対応が望ましい:
  条件:
    - 複数PBIに影響する
    - 作業効率が大幅に低下する
    - 品質に重大な影響がある
  
  対応:
    - 週次レビューで優先的に議論
    - 2週間以内に対応開始

Medium（中）- 計画的に対応:
  条件:
    - 特定のPBIに影響する
    - 作業効率が軽微に低下する
    - 改善の余地がある
  
  対応:
    - 月次レビューで議論
    - 1-3ヶ月以内に対応

Low（低）- 改善アイデアとして記録:
  条件:
    - 現状でも問題なく作業可能
    - より良い方法がある（必須ではない）
    - 将来的な改善案
  
  対応:
    - 四半期レビューで議論
    - 優先度が高い課題の後に対応
```

---

### 優先度判定フローチャート

```
課題発見
  ↓
[Q1] 作業が完全にブロックされるか？
  Yes → Critical
  No  → Q2へ
  ↓
[Q2] セキュリティ・データ損失リスクがあるか？
  Yes → Critical
  No  → Q3へ
  ↓
[Q3] 複数PBIに影響するか？
  Yes → High
  No  → Q4へ
  ↓
[Q4] 作業効率が大幅に低下するか？
  Yes → High
  No  → Q5へ
  ↓
[Q5] 特定のPBIに影響するか？
  Yes → Medium
  No  → Low
```

## 課題の追跡とレビュー

### 記録場所

```yaml
課題管理ファイル:
  場所: docs/ai-issues/
  
  ファイル構成:
    - README.md: 課題管理の概要
    - open/: 未対応の課題
      - critical/: Critical優先度
      - high/: High優先度
      - medium/: Medium優先度
      - low/: Low優先度
    - in-progress/: 対応中の課題
    - resolved/: 対応完了の課題
    - closed/: クローズした課題
  
  命名規則:
    AI-ISSUE-YYYYMMDD-NNN-[簡潔なタイトル].md
    例: AI-ISSUE-20251119-001-document-contradiction.md
```

---

### レビュー頻度

```yaml
日次レビュー:
  対象: Critical優先度の課題
  参加者: Tech Lead
  内容: 新規Critical課題の確認、即座に対応

週次レビュー:
  対象: Critical + High優先度の課題
  参加者: Tech Lead + AI担当者
  内容:
    - 新規課題の確認
    - 対応中課題の進捗確認
    - 対応優先順位の調整

月次レビュー:
  対象: 全優先度の課題
  参加者: Engineering Team全体
  内容:
    - 課題の傾向分析
    - 対応完了課題の効果測定
    - プロセス改善の検討
    - Medium優先度課題の対応計画

四半期レビュー:
  対象: 全課題 + 統計分析
  参加者: Engineering Leadership
  内容:
    - 四半期の課題傾向分析
    - 大規模な改善計画の策定
    - AIの改善提案の評価
    - Low優先度課題の対応判断
```

---

### レビュー時の判断基準

```yaml
課題のクローズ判断:
  
  対応完了:
    - 改善実施完了
    - 効果測定完了
    - ドキュメント更新完了
    → Closed（クローズ）
  
  重複:
    - 既存課題と重複
    → Closed（重複として）
  
  対応不要:
    - 状況変化により問題解消
    - コスト > 効果
    → Closed（対応不要として）
  
  保留:
    - 現時点で対応困難
    - 他の課題が優先
    → Open（優先度Low）
```

## AIエージェントへの指示

```yaml
ai_guidelines:
  課題発見時:
    1. 問題の性質を判断（カテゴリ・優先度）
    2. 適切なタイミングで記録（即座/Phase完了時/PBI完了時）
    3. テンプレートに従って詳細を記録
    4. 必要に応じて人間にエスカレーション
  
  課題記録の頻度:
    - Critical: 発見次第即座に記録
    - High: Phase完了時に記録
    - Medium/Low: PBI完了時にまとめて記録
  
  ai_must_do:
    - "✓ 問題を発見したら必ず記録"
    - "✓ テンプレートに従う"
    - "✓ 具体的な改善案を含める"
    - "✓ 優先度を適切に判断"
    - "✓ Critical課題は即座にエスカレーション"
  
  ai_should_not:
    - "✗ 問題を発見しても記録しない"
    - "✗ 曖昧な記述で記録"
    - "✗ 改善案なしで問題だけ指摘"
    - "✗ 優先度の判断を怠る"
  
  # === 絶対禁止事項 ===
  ai_must_never_do:
    - "🚨 組織標準ドキュメント（/organization-standards/配下）の直接修正"
    - "🚨 タイポ・誤字・リンク切れの直接修正"
    - "🚨 軽微な改善の直接実施"
    - "🚨 組織標準ドキュメントの削除・移動・リネーム"
    - "🚨 組織標準ドキュメントの新規作成"
  
  correct_process:
    - "✓ 問題発見 → 課題起票 → 人間がレビュー・承認 → 人間が修正"
    - "✓ 課題起票時にdiffスタイルで修正案を提案"
    - "✓ AI-DOCUMENT-MODIFICATION-POLICY.mdを参照"

記録のベストプラクティス:
  - 具体的に: 「ドキュメントが不明確」ではなく「XXドキュメントのセクション3.2でYYの記載が欠落」
  - 改善案を含める: 問題指摘だけでなく、具体的な改善案を提示
  - 定量的に: 「遅い」ではなく「3分かかる、1分に短縮可能」
  - 実行可能に: 改善案は実現可能な具体的内容
  - 修正は人間が実施: AIは提案のみ、実装は人間
```

## 📋 基本情報

---

## 🔍 課題の詳細

### 発見した状況
Phase 4でPRを作成後、人間レビュー待ちで平均4時間待機しています。

### 具体的な問題内容
- リファクタリング（REF）の軽微な変更でも人間レビュー必須
- CI/CD全パス、カバレッジ90%でも承認待ち
- 変更量が少ない（3ファイル、50行）でも同様

### 現状の影響
- **作業への影響**: 遅延（PR待ち時間が4時間）
- **頻度**: 頻繁（REFタイプのPBIで毎回）
- **影響範囲**: プロジェクト全体

---

## 💡 改善提案

### 提案内容
条件付き自動承認ルールの導入:
- 条件: REF/BUG/NFDタイプ、変更量少、CI/CD全パス、カバレッジ>=85%
- 対応: 自動承認後、日次サマリーで事後報告

### 期待される効果
- **時間削減**: 4時間 → 5分（95%削減）
- **品質維持**: CI/CDで品質担保、事後確認で監視
- **効率化**: 人間は複雑な変更に集中

### 実装の難易度
- **難易度**: 中
- **推定工数**: 2日（ガイドライン策定、自動化実装）

---

## ⚙️ プロセス詳細情報

- **プロセス名**: Phase 4 レビュー・QA
- **Phase**: Phase 4
- **関連ドキュメント**: phase-4-review-qa-guide.md

1. PRを作成
2. CI/CD実行
3. 人間レビュー待ち（平均4時間）
4. 承認
5. マージ

### 提案する改善手順
1. PRを作成
2. CI/CD実行
3. **自動承認条件チェック**
4. **条件満たせば自動マージ（5分）**
5. **日次サマリーで事後報告**

### 定量的な効果見込み
- **現状の所要時間**: 4時間
- **改善後の所要時間**: 5分
- **削減時間**: 3時間55分 (98%)
