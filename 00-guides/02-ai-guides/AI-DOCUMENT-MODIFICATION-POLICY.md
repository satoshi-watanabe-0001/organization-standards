## 基本原則

### 🚨 **最重要ルール**

```yaml
絶対禁止:
  AIによる組織標準ドキュメントの直接修正は絶対に禁止

理由:
  - 組織全体に影響する
  - 複数プロジェクトに影響する
  - 品質保証プロセスが必要
  - 人間のレビュー・承認が必須

正しい手順:
  1. 問題を発見
  2. 課題起票（AI-ISSUE-TRACKING-PROCESS.mdに従う）
  3. 人間がレビュー・承認
  4. 人間が修正実施
```

## 組織標準ドキュメントの定義

### 組織標準ドキュメントとは

以下のディレクトリ配下にある全てのドキュメント:

```yaml
組織標準ドキュメント（修正絶対禁止）:
  
  1. /organization-standards/ 配下の全ドキュメント:
     - 00-guides/
     - 01-coding-standards/
     - 02-architecture-standards/
     - 03-development-process/
     - 04-quality-standards/
     - 05-technology-stack/
     - 06-tools-and-environment/
     - 07-security-compliance/
     - 08-templates/
     - 09-reference/
     - 10-governance/
     - README.md
     - その他全ての .md ファイル

特徴:
  - 組織全体で共通使用
  - 複数プロジェクトに影響
  - 厳格な品質管理が必要
  - 変更には人間の承認が必須
```

## 絶対禁止事項

### ❌ AIが絶対にやってはいけないこと

```yaml
禁止事項リスト:

1. ❌ 組織標準ドキュメントの直接修正
   - タイポ・誤字の修正も禁止
   - 軽微な表記揺れの修正も禁止
   - 例の追加も禁止
   - リンク切れの修正も禁止
   - フォーマット調整も禁止

2. ❌ 組織標準ドキュメントの削除
   - ファイル削除は絶対禁止
   - セクション削除も禁止

3. ❌ 組織標準ドキュメントの新規作成
   - 新しい標準ドキュメントの作成は禁止
   - テンプレートの追加も禁止

4. ❌ 組織標準ドキュメントの移動・リネーム
   - ファイルの移動は禁止
   - ファイル名の変更は禁止

理由:
  全ての変更は組織全体に影響するため、
  人間のレビュー・承認プロセスが必須
```

---

### 具体例: AIがやってはいけないこと

```markdown
❌ 禁止例1: タイポ修正
phase-3-implementation-guide.md に以下のタイポを発見:
  "テストカバレッヂ" → "テストカバレッジ"

AIがやってはいけないこと:
  - ファイルを直接修正する
  - git commit を作成する
  - PRを作成する

AIがやるべきこと:
  - 課題を起票（AI-ISSUE-TRACKING-PROCESS.mdに従う）
  - 優先度: Low
  - カテゴリ: ドキュメント
  - 人間の修正を待つ
```

```markdown
❌ 禁止例2: リンク切れ修正
README.md に以下のリンク切れを発見:
  [Phase 3ガイド](phase-3-guide.md) 
  → 実際のファイル名は phase-3-implementation-guide.md

AIがやってはいけないこと:
  - README.md を直接修正する

AIがやるべきこと:
  - 課題を起票
  - 優先度: Medium
  - カテゴリ: ドキュメント
  - 正しいリンクを提案
```

```markdown
❌ 禁止例3: 情報の追加
testing-strategy.md に情報が不足していることを発見:
  "ユニットテストの目標カバレッジ: 80%以上"
  → "ビジネスロジック: 90%以上"の記載がない

AIがやってはいけないこと:
  - testing-strategy.md に情報を追加する

AIがやるべきこと:
  - 課題を起票
  - 優先度: High（作業に影響）
  - カテゴリ: ドキュメント
  - 追加すべき情報を具体的に提案
```

## AIが実施すべきこと

### ✅ 正しいプロセス

```yaml
ステップ1: 問題の発見
  - 作業中に組織標準ドキュメントの問題を発見

ステップ2: 課題起票
  - AI-ISSUE-TRACKING-PROCESS.md に従って課題を記録
  - 必須テンプレートを使用
  - カテゴリ: ドキュメント
  - 優先度を判定

ステップ3: 具体的な修正案を提案
  - 現状の問題箇所（ファイル名、行番号）
  - 提案する修正内容（diffスタイル）
  - 修正理由
  - 影響範囲

ステップ4: 作業を継続
  - 課題起票後、現状のドキュメントに従って作業を継続
  - 人間の修正を待つ必要はない
  - 必要に応じてワークアラウンドを使用

ステップ5: 報告
  - 優先度に応じた頻度で報告
  - Critical: 即座
  - High: Phase完了時
  - Medium/Low: PBI完了時
```

## 🔧 標準リポジトリ保守時の例外条件

### 例外が適用される条件

通常、AIによる組織標準ドキュメントの直接修正は禁止されていますが、以下の**すべての条件**を満たす場合に限り、例外として修正が許可されます：

```yaml
標準リポジトリ保守の例外条件:

  必須条件（すべて満たすこと）:
    1. 明示的な依頼:
       - 人間（ドキュメント管理者、Tech Lead等）からの明示的な修正依頼がある
       - 「ドキュメントを改善してください」「リンク切れを修正してください」等の具体的指示
    
    2. PRによるレビュー:
       - 直接mainブランチにpushしない
       - 必ずPRを作成し、人間のレビュー・承認を経る
       - PRの説明に変更内容と理由を明記
    
    3. 変更範囲の限定:
       - 依頼された範囲のみを修正
       - 関連しない箇所は修正しない
       - 大規模な構造変更は事前に確認を取る
    
    4. 記録の保持:
       - 変更履歴をコミットメッセージに明記
       - PRの説明に変更理由を記載
       - 必要に応じてバージョン履歴を更新

  例外が適用されないケース:
    - AIが自主的に発見した問題の修正（→ 課題起票が必要）
    - 明示的な依頼なしでの「改善」
    - PRを経由しない直接修正
    - 依頼範囲を超えた修正
```

### 例外適用時の判断フロー

```
組織標準ドキュメントを修正したい
  ↓
[Q1] 人間からの明示的な修正依頼があるか？
  No  → ❌ 修正禁止、課題起票
  Yes → Q2へ
  ↓
[Q2] PRを作成してレビューを受けるか？
  No  → ❌ 修正禁止
  Yes → Q3へ
  ↓
[Q3] 依頼された範囲内の修正か？
  No  → ⚠️ 範囲外は課題起票、範囲内のみ修正
  Yes → ✅ 修正可能（PRでレビュー必須）
```

### 例外適用時の注意事項

- **PRの説明には必ず以下を含める**:
  - 依頼者（誰からの依頼か）
  - 変更理由（なぜ修正が必要か）
  - 変更内容の概要（何を修正したか）
  - 影響範囲（どのドキュメント・機能に影響するか）

- **レビュー時の確認ポイント**:
  - 依頼範囲内の修正か
  - 既存の整合性を壊していないか
  - 新たなリンク切れを生んでいないか

## 例外（修正可能なドキュメント）

### ✅ AIが修正可能なドキュメント

以下のドキュメントは**AIが直接修正可能**:

```yaml
修正可能なドキュメント:

1. 個別PBI専用ドキュメント:
   - docs/requirements/PROJ-1234-requirements.md
   - docs/design/PROJ-1234-detailed-design.md
   - docs/api/pbi/PROJ-1234-api-contract.yaml
   
   特徴:
     - 特定のPBIにのみ関連
     - 他のPBIに影響しない
     - プロジェクト固有

2. AIの判断記録ファイル:
   - docs/ai-decisions/document-conflicts-resolved.md
   - docs/ai-decisions/auto-approved-prs.md
   - docs/ai-decisions/technology-selections.md
   
   特徴:
     - AIの自律判断を記録
     - 組織標準ではない

3. AI課題管理ファイル:
   - docs/ai-issues/open/*/AI-ISSUE-*.md
   - docs/ai-issues/in-progress/AI-ISSUE-*.md
   - docs/ai-issues/resolved/AI-ISSUE-*.md
   
   特徴:
     - AIが作成・管理
     - 課題の追跡用

4. プロジェクト固有のドキュメント:
   - プロジェクトリポジトリ内のドキュメント
   - 組織標準以外のドキュメント
   
   条件:
     - /organization-standards/ 配下でない
     - 組織全体に影響しない

判断基準:
  - ファイルパスに /organization-standards/ が含まれるか？
    Yes → 修正絶対禁止
    No → 修正可能（ただし慎重に）
```

---

### 判断フローチャート

```
ドキュメントを修正したい

[Q1] ファイルパスに /organization-standards/ が含まれるか？

  No  → Q2へ
  ↓
[Q2] 特定のPBIにのみ関連するドキュメントか？
  Yes → ✅ 修正可能
  No  → Q3へ
  ↓
[Q3] AIの判断記録ファイルか？
  Yes → ✅ 修正可能
  No  → Q4へ
  ↓
[Q4] AI課題管理ファイルか？
  Yes → ✅ 修正可能
  No  → ⚠️ 慎重に判断、不明なら課題起票
```

## 違反時の対応

### 違反検知

```yaml
違反パターン:
  1. AIが組織標準ドキュメントを直接修正
  2. AIが組織標準ドキュメントを削除
  3. AIが組織標準ドキュメントを新規作成
  4. AIが組織標準ドキュメントを移動・リネーム

検知方法:
  - git commit履歴のレビュー
  - PRのファイル変更確認
  - CI/CDでの自動チェック
```

---

### 違反時のアクション

```yaml
即座の対応:
  1. 該当のcommit/PRをリバート
  2. インシデントレポート作成
  3. AIにフィードバック
  4. このポリシーの再確認

根本原因分析:
  - なぜAIが直接修正したか
  - ドキュメントの問題があったか
  - プロンプトの問題があったか

再発防止策:
  - AIガイドラインの強化
  - CI/CDでの自動チェック追加
  - 定期的なレビュー
```
