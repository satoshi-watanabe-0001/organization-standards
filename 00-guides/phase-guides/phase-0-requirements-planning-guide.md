# Phase 0: 要件分析・企画ガイド

> PBIから実装可能なタスクへの変換 - 軽量ナビゲーションガイド

**対象**: Devin、Cursor、その他の自律型AIエージェント  
**前提**: JIRA PBI (Product Backlog Item) が既に作成されている  
**目的**: PBIを理解し、実装可能なタスクと要件分析書に変換する

---

## 📋 このガイドについて

### ドキュメントタイプ
- **軽量ナビゲーション型**: 既存ドキュメントへの参照を重視
- **所要時間**: 5-10分で読める
- **重複回避**: 詳細は既存ドキュメントを参照

### Phase 0 の位置づけ

```
[JIRA PBI] 
    ↓
┌─────────────────────────────────────────┐
│  Phase 0: 要件分析・企画 ★ここ          │
│  - PBI理解 → 要件抽出 → タスク分解       │
└─────────────────────────────────────────┘
    ↓
[要件分析書 + タスクリスト]
    ↓
Phase 1 以降へ（状況に応じてスキップ可能）
```

### スキップ可能性
⚠️ **Phase 0 は原則としてスキップ不可**
- どんなPBIでも最低限の要件確認は必須
- **簡易版での実施は可能**:
  - バグ修正: 15-30分の簡易版
  - 緊急修正: 必要最小限の確認
  - 新規プロジェクト: 1-2時間のフル版

---

## ⚠️ 禁止事項チェック

### このフェーズ特有の禁止事項
- **要件確認不足**: PBIを十分に理解せずにタスク分解を進めてはいけない
- **曖昧なまま放置**: 不明点があるのに確認せず推測で進めてはいけない
- **勝手な要件追加**: PBIに記載のない機能を独断で追加してはいけない
- **スコープ拡大**: 必要最小限の実装範囲を超えて勝手に機能を拡張してはいけない

### 全フェーズ共通禁止事項
- **CI/CD設定の無断変更**: 明示的な指示がない限り、GitHub Actions、CircleCI、Jenkins等のCI/CD設定ファイルを変更してはいけない
- **本番環境への直接変更**: 本番データベース、本番サーバー、本番設定ファイルを直接変更してはいけない
- **セキュリティ設定の独断変更**: 認証・認可・暗号化等のセキュリティ設定を独自判断で変更してはいけない
- **組織標準外技術の無断導入**: プロジェクトで使用していない新しいライブラリ・フレームワーク・言語を独断で導入してはいけない
- **プロジェクト構造の大幅変更**: ディレクトリ構造、命名規則、アーキテクチャパターンを独断で大きく変更してはいけない

### 📚 詳細確認
禁止事項の詳細、具体例、例外ケースについては以下のドキュメントを参照してください：
- [AI開発タスクの禁止事項](../01-organization-standards/ai-task-prohibitions.md)
- [禁止事項チェックリスト](../01-organization-standards/ai-task-prohibitions-checklist.md)

## 📊 Phase 0 の所要時間

| PBI複雑度 | 所要時間 | 例 |
|----------|---------|-----|
| **簡易** | 15-30分 | バグ修正、小規模改善 |
| **標準** | 30-60分 | 単一機能追加 |
| **複雑** | 1-2時間 | 複数機能、設計変更 |
| **超大規模** | 2-4時間 | 新規プロジェクト |

## 🔄 Phase 0 のワークフロー: 7ステップ

### **Step 0.1: PBI読み込みと構造理解** (5-10分)

**実行内容**:
1. PBI全体を読み込む（タイトル、説明、受入基準、制約）
2. PBIタイプを判別（7タイプ）:
   - 新規プロジェクト / 既存への機能追加 / バグ修正
   - リファクタリング / 設計変更 / Hotfix / 実験的機能

**チェックリスト**:
- [ ] PBIの基本情報を把握
- [ ] PBIタイプを判別
- [ ] 優先度・期限を確認
- [ ] 関連Issueを確認

**参照**: なし（PBIを直接読む）

---

### **Step 0.2: 情報抽出と不明点の特定** (10-15分)

**実行内容**:
1. 機能要件を抽出（何を実装するか）
2. 非機能要件を抽出（パフォーマンス、セキュリティ、スケーラビリティ）
3. 技術的制約を把握
4. 不明点を特定し、重要度を判断

**不明点の処理**:
- 🔴 重要（開発に影響） → 即座に人間に確認要求
- 🟡 軽微 → 不明点リストに記録、標準的な想定で進行

**チェックリスト**:
- [ ] 機能要件を抽出・整理
- [ ] 非機能要件を抽出・整理
- [ ] 技術的制約を把握
- [ ] 不明点を特定
- [ ] 確認リクエストを作成（必要な場合）

**参照**:
- 🟡 ../../09-reference/glossary.md（技術用語の確認）

**テンプレート**:
- 🟡 clarification-request-template（不明点確認用）

---

### **Step 0.3: タスク細分化** (20-40分)

**実行内容**:
1. PBIをFeature → Task → Sub-Taskに分解
2. 各タスクの所要時間を見積もる
3. 優先度を設定
4. MECE（漏れなく、重複なく）を確認

**タスク粒度の基準**:
- **Feature**: 1日〜数日（100-300行）
- **Task**: 数時間〜1日（20-50行）
- **Sub-Task**: 30分〜数時間（5-20行）

**チェックリスト**:
- [ ] タスクを適切な粒度に分解
- [ ] 各タスクの所要時間を見積もり
- [ ] 優先度を設定
- [ ] タスク分解リストを作成（Markdown or JSON）
- [ ] MECEを確認

**参照**: なし（PBIの内容から分解）

**出力フォーマット**:
- Markdown Table形式（推奨）
- JSON形式（ツール連携用）

---

### **Step 0.4: 依存関係の特定と可視化** (10-15分)

**実行内容**:
1. タスク間の依存関係を特定
2. 依存関係マップを作成（Mermaid or テキストツリー）
3. クリティカルパスを特定
4. 並行実行可能タスクを特定

**依存関係の種類**:
- **技術的依存**: 実装が別の成果物を必要
- **データ依存**: データ構造に依存
- **機能的依存**: 論理的な順序
- **リソース依存**: 同じリソースを使用

**チェックリスト**:
- [ ] 依存関係を特定
- [ ] 依存関係マップを作成
- [ ] クリティカルパスを特定
- [ ] 並行実行可能タスクを特定
- [ ] 循環依存がないことを確認

**参照**: なし

---

### **Step 0.5: 初期技術方針の決定** (10-15分)

**実行内容**:
1. 組織標準技術を確認
2. PBIの制約を考慮
3. 概要レベルの技術方針を決定（詳細はPhase 1）

**決定すべき項目**（概要レベル）:
- プログラミング言語・フレームワーク
- データベース種類
- 認証・認可方式（該当する場合）
- API設計スタイル

**チェックリスト**:
- [ ] 組織標準を確認
- [ ] PBIの制約を反映
- [ ] 初期技術方針を記録
- [ ] Phase 1で決定すべき項目を明確化

**参照**:
- 🔴 ../../05-technology-stack/approved-technologies.md（組織標準）
- ⚪ ../../10-governance/technology-radar.md（技術評価）

---

### **Step 0.6: 要件分析書の作成** (15-20分)

**実行内容**:
1. テンプレートを使用して要件分析書を作成
2. Step 0.1〜0.5の成果物を統合
3. 未解決事項・確認待ち項目を記載

**要件分析書の主要セクション**:
1. プロジェクト情報
2. 背景と目的
3. 機能要件
4. 非機能要件
5. 技術的制約
6. 受入基準
7. タスク分解（参照）
8. 初期技術方針
9. 依存関係とクリティカルパス
10. 未解決事項
11. リスク分析
12. 見積もり

**チェックリスト**:
- [ ] requirements-analysis-[PBI-KEY].md を作成
- [ ] 全セクションが記入されている
- [ ] タスク分解リストへのリンクがある
- [ ] 不明点確認リストへのリンクがある（必要な場合）

**参照**:
- 🟡 ../../08-templates/project-proposal-template.md
- 🟡 ../../08-templates/（その他テンプレート）

---

### **Step 0.7: Phase 0 完了確認** (5分)

**実行内容**:
1. 完了チェックリストを実行
2. 成果物を確認
3. 次フェーズを判断

**完了チェックリスト**:
- [ ] PBIの全内容を理解した
- [ ] 要件を抽出・整理した
- [ ] タスクを適切に分解した
- [ ] 依存関係を特定した
- [ ] 初期技術方針を決定した
- [ ] 要件分析書を作成した
- [ ] タスク分解リストを作成した
- [ ] 不明点が解消された（または確認済み）

**成果物確認**:
- [ ] requirements-analysis-[PBI-KEY].md が存在
- [ ] task-breakdown-[PBI-KEY].md が存在
- [ ] clarification-request-[PBI-KEY].md が存在（必要な場合）

**参照**:

## 🎯 次フェーズの判断フロー ⭐重要

### Phase 0 完了後の判断

**全フェーズを実行する必要はありません。PBIの内容に応じて適切なフェーズから開始してください。**

```
[Q1] 新規プロジェクトか?
  Yes → Phase 1 へ（プロジェクト初期化が必要）
  No  → Q2 へ

[Q2] プロジェクト構造・技術スタック確定済みか?
  No  → Phase 1 へ（技術スタック選定が必要）
  Yes → Phase 2 へ（Phase 1 スキップ）

[Q3] 設計変更が必要か?
  Yes → Phase 2 へ（設計フェーズが必要）
  No  → Phase 3 へ（Phase 2 スキップ）

[Q4] コード変更が必要か?
  Yes → Phase 3 へ（実装フェーズが必要）
  No  → Phase 4 へ（Phase 3 スキップ）
```

### 典型的なフロー例

| PBIタイプ | 推奨フロー |
|----------|-----------|
| **新規プロジェクト** | Phase 0 → 1 → 2 → 3 → 4 → 5 → 6（全フェーズ） |
| **既存への機能追加** | Phase 0 → 2 → 3 → 4 → 5（Phase 1 スキップ） |
| **バグ修正** | Phase 0（簡易版）→ 3 → 4（Phase 1,2 スキップ） |
| **リファクタリング** | Phase 0 → 3 → 4（Phase 1,2 スキップ） |
| **設計変更** | Phase 0 → 2 → 3 → 4 → 5（Phase 1 スキップ） |
| **Hotfix** | 簡易Phase 0 → 3 → 簡易Phase 4 → 5 |
| **実験的機能** | 簡易Phase 0 → 3（Phase 1,2,4,5,6 スキップ） |

### スキップ時の記録

```markdown
## フェーズ実行記録: [PBI-KEY]

### 実行したフェーズ
- ✅ Phase 0: 要件分析（実施）
- ⏭️ Phase 1: プロジェクト初期化（スキップ - 既存プロジェクト）
- ✅ Phase 2: 設計（実施）
- ✅ Phase 3: 実装（実施）
- ✅ Phase 4: レビュー（実施）
- ✅ Phase 5: デプロイ（実施）
- ⏭️ Phase 6: 運用・保守（後日実施）

### スキップ理由
- Phase 1: 既存プロジェクト「ProjectX」への機能追加
- Phase 6: 新機能のため運用実績なし。1週間後に実施予定
```

**詳細**: AI-MASTER-WORKFLOW-GUIDE.md の「🎯 フェーズ実行の判断ガイド」

## 🚨 エスカレーション基準

### 即座にエスカレーション（人間判断が必要）
以下の状況では、作業を中断して人間に確認:

## 🔧 トラブルシューティング

### 問題1: PBIの受入基準が不明確
**解決策**:
1. 不明点確認リクエストを作成（Step 0.2）
2. 標準的な受入基準を想定（AI-QUICK-REFERENCE.md参照）
3. 人間に確認を依頼

### 問題2: タスク分解の粒度が分からない
**解決策**:
1. 「1つのGitコミットで完結できる」を基準に
2. 所要時間: 数時間〜1日のタスクを目指す
3. 過去の類似PBIを参考に

### 問題3: 技術スタックが不明
**解決策**:
1. ../../05-technology-stack/approved-technologies.md を確認
2. 既存プロジェクトの技術スタックを参照
3. Phase 1 で詳細決定することを記録

### 問題4: 依存関係が複雑で整理できない
**解決策**:
1. まずは明確な依存関係のみ記録
2. 不明確な部分は Phase 1, 2 で再検討
3. シンプルな図から始める（複雑化しない）
