# Phase 5 作業ブロック チェックリスト

## 📋 チェックリストの使い方

このチェックリストは、**Phase 5の各作業ブロック実施時**に使用します。

### 適用タイミング
- マイルストーン完了後のPhase 5作業ブロック開始時
- Phase 4（レビュー・QA）期間中の並行作業時
- Phase 5最終化ブロック実施時

### チェックの進め方
1. **事前準備**セクションを作業開始前に確認
2. **作業中**セクションを作業中に随時確認
3. **完了確認**セクションを作業終了時に確認

---

## ✅ 事前準備チェック（作業開始前）

### 実装完了の確認

```yaml
implementation_status:
  - [ ] 対象マイルストーンの実装がすべて完了している
  - [ ] すべてのユニットテストがパスしている
  - [ ] 統合テストが完了している
  - [ ] コードレビューが完了している
  - [ ] コードがmainブランチにマージされている
```

---

### 環境準備

```yaml
workspace:
  - [ ] 実装コードをローカルに最新版でpullしている
  - [ ] ドキュメント作成用のエディタを起動している
  - [ ] 図作成ツールを起動している
      - Mermaid / PlantUML / Draw.io / dbdiagram.io 等
  - [ ] テンプレートファイルを準備している
  - [ ] 前回のPhase 5作業ブロックの成果物を確認している
```

---

### 時間確保

```yaml
schedule:
  - [ ] 予定時間（4-6時間）を確保している
  - [ ] 会議がブロックされている
  - [ ] 通知をオフにしている（Slack, Email等）
  - [ ] チームメンバーに作業中であることを伝えている
```

---

### 参照資料の準備

```yaml
references:
  - [ ] Phase 2A成果物を手元に準備
      - API契約書
      - 制約条件文書
      - ADR
  - [ ] 実装コードのディレクトリ構造を確認
  - [ ] コミット履歴を確認可能にしている
  - [ ] 実装中の決定事項メモを確認
```

## 🔧 作業中チェック

### 成果物別チェックリスト

#### API仕様書作成

```yaml
api_specification:
  content:
    - [ ] エンドポイント一覧が完全である
    - [ ] リクエスト・レスポンスのスキーマが正確
    - [ ] 認証・認可の仕様が記載されている
    - [ ] エラーレスポンスが網羅されている
    - [ ] レート制限の仕様が明記されている
  
  quality:
    - [ ] 実装コードと完全に一致している
    - [ ] サンプルリクエスト・レスポンスを記載
    - [ ] OpenAPI形式（Swagger）でエクスポート可能
  
  tools:
    - [ ] Swagger Editorで検証済み
    - [ ] Postmanコレクションを作成（推奨）
```

---

#### アーキテクチャ図作成

```yaml
architecture_diagrams:
  required_diagrams:
    - [ ] システム全体構成図
    - [ ] コンポーネント図
    - [ ] シーケンス図（主要フロー）
    - [ ] デプロイメント図
  
  quality:
    - [ ] 図が実装と一致している
    - [ ] 矢印の向きが正しい
    - [ ] ラベルが明確で読みやすい
    - [ ] 凡例が記載されている
  
  format:
    - [ ] Mermaid形式（推奨）または PNG/SVG
    - [ ] ソースコード（.mmd）も保存
    - [ ] バージョン管理に含める
```

---

#### データモデル文書作成

```yaml
data_model:
  content:
    - [ ] ER図が作成されている
    - [ ] すべてのテーブル定義が記載されている
    - [ ] インデックス設計が記載されている
    - [ ] 外部キー制約が図示されている
    - [ ] マイグレーション履歴が記録されている
  
  quality:
    - [ ] 実装スキーマと完全に一致
    - [ ] データ型が正確
    - [ ] 制約（NOT NULL, UNIQUE等）が正確
  
  tools:
    - [ ] dbdiagram.io / ERDiagram等で作成
    - [ ] DDL（CREATE TABLE）を含める
    - [ ] サンプルデータを記載（推奨）
```

---

#### 設計書作成

```yaml
design_document:
  sections:
    - [ ] 概要・目的が記載されている
    - [ ] アーキテクチャの説明が完全
    - [ ] 主要コンポーネントの説明
    - [ ] 技術スタックの説明
    - [ ] セキュリティ設計
    - [ ] パフォーマンス設計
    - [ ] 運用設計
  
  quality:
    - [ ] 新規参画者が理解できる内容
    - [ ] 図とテキストが整合している
    - [ ] 専門用語に注釈を付けている
```

---

#### ADR（Architecture Decision Record）更新

```yaml
adr:
  - [ ] 実装中に追加された技術決定を記録
  - [ ] 決定の背景・理由が明確
  - [ ] 代替案との比較が記載されている
  - [ ] 決定日と決定者が記録されている
  - [ ] ADRテンプレートに準拠している
```

---

### 作業時の注意事項

```yaml
working_guidelines:
  focus:
    - [ ] ドキュメント作成に集中（実装には戻らない）
    - [ ] 80%ルール: 80%完成度で次に進む
  
  quality:
    - [ ] 実装コードを参照しながら作業
    - [ ] コミット履歴から設計判断の理由を確認
    - [ ] 不明点はコードレビューコメントを確認
  
  efficiency:
    - [ ] AI支援ツールを活用（OpenAPI生成、図の自動生成等）
    - [ ] テンプレートを最大限活用
    - [ ] 完璧を求めすぎない
```

## ✅ 完了確認チェック（作業終了時）

### 成果物の品質確認

```yaml
quality_assurance:
  completeness:
    - [ ] 計画した成果物がすべて作成されている
    - [ ] 各成果物が80%以上完成している
    - [ ] 必須セクションがすべて埋まっている

    - [ ] 実装コードと一致している

  readability:
    - [ ] 第三者が理解できる内容である
    - [ ] 図が見やすい
    - [ ] 用語が統一されている
```

## ⚠️ よくある問題と対処法

### 問題1: 時間が足りない

**症状**: 予定時間内に終わらない

**対処法**:
```yaml
solutions:
  - 80%ルールを適用（完璧を求めない）
  - 優先度の低い成果物を次のブロックに延期
  - スコープを縮小（最小限の情報に絞る）
  - AIツールで効率化（OpenAPI生成、図の自動生成）
```

---

### 問題2: 実装との差異を発見

**症状**: ドキュメント作成中に実装の誤りを発見

**対処法**:
```yaml
solutions:
  - Issueを起票して記録
  - 緊急度が高い場合は実装修正を優先
  - ドキュメントには現状を記録（as-built）
  - Phase 5完了後に修正対応
```

---

### 問題3: 情報が不足している

**症状**: ドキュメント化に必要な情報が不明

**対処法**:
```yaml
solutions:
  - コードレビューコメントを確認
  - コミット履歴を調査
  - 実装者にSlackで質問（短時間で）
  - 不明点を明記して次のブロックで解決
```

---

### 問題4: 図の作成に時間がかかる

**症状**: アーキテクチャ図の作成に予想以上の時間

**対処法**:
```yaml
solutions:
  - 手書きのスケッチ → 写真撮影 → 後で清書
  - Mermaidでシンプルな図を作成（コードベース）
  - AIツールで自動生成してから修正
  - 最小限の図で済ませる（詳細は次回）
```
