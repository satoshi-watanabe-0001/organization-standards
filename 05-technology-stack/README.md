# 技術スタック標準 / Technology Stack Standards

**最終更新日**: 2025-10-24  
**バージョン**: 1.0.0  
**対象**: アーキテクト、テクニカルリード、自律型AI Devin  
**適用範囲**: 全プロジェクト共通技術選定基準

---

## 📖 概要

このディレクトリは、組織全体で承認された技術スタックと選定基準を定義し、一貫性のある技術選定を支援します。特に自律型AI Devinが適切な技術を選択し、組織標準に沿ったシステムを構築できるよう、明確なガイドラインを提供します。

### 🎯 目的

- **技術の標準化**: 承認済み技術スタックの明確化
- **選定の効率化**: 技術選定基準による迅速な意思決定
- **リスクの低減**: 実績のある技術の採用による安定性確保
- **スキルの集約**: 限定された技術スタックによるチーム専門性の向上

## 📋 各ファイルの役割

### `approved-technologies.md` - 承認済み技術一覧 🔴 Tier 1
**内容**:
- プログラミング言語（TypeScript、Python、Java等）
- フロントエンドフレームワーク（React、Vue.js等）
- バックエンドフレームワーク（Express、FastAPI等）
- 承認プロセス
- バージョンポリシー
- 非推奨技術リスト

**対象読者**: 全開発者、アーキテクト、Devin AI

**利用シーン**:
- プロジェクト開始時の技術選定
- 新技術導入検討時
- レガシー技術からの移行計画時

**重要度**: 🔴 **必須** - すべての技術選定の基準

**承認済み技術の分類**:
```
Tier 1: 推奨（デフォルトで採用）
Tier 2: 承認済み（特定の用途で採用可）
Tier 3: 評価中（パイロットプロジェクトのみ）
非推奨: 新規採用禁止
```

### `database-stack.md` - データベース技術スタック 🔴 Tier 1
**内容**:
- RDBMS選定基準（PostgreSQL、MySQL等）
- NoSQL選定基準（MongoDB、Redis等）
- データベースバージョンポリシー
- データベース接続ライブラリ
- ORMとクエリビルダー
- マイグレーションツール

**対象読者**: データベースエンジニア、バックエンド開発者、Devin AI

**利用シーン**:
- データベース技術選定時
- データモデリング時
- パフォーマンスチューニング時

**重要度**: 🔴 **必須** - データベースを使用するすべてのプロジェクト

**推奨データベース**:
```
RDBMS:
  - PostgreSQL 14+ (推奨)
  - MySQL 8.0+ (レガシーシステム互換)

NoSQL:
  - Redis 7+ (キャッシュ)
  - MongoDB 6+ (ドキュメント指向)

ORM:
  - Prisma (TypeScript/Node.js)
  - SQLAlchemy (Python)
  - Hibernate (Java)
```

## 🤖 Devin（自律型AI）の利用パターン

### パターン1: 新規プロジェクト技術選定時
```
1. approved-technologies.md で承認済み技術を確認
2. プロジェクト要件に応じて最適な技術を選択
3. database-stack.md でデータベース技術を選定
4. cloud-providers.md でクラウドプロバイダーを選定
5. 技術スタックドキュメントを作成
```

**プロンプト例**:
```
以下の技術スタック標準に基づいて、プロジェクトの技術選定を行ってください：
/05-technology-stack/approved-technologies.md
/05-technology-stack/database-stack.md

プロジェクト要件：
- Webアプリケーション
- RESTful API提供
- リアルタイム通知機能
- 想定ユーザー数: 10万人
- データ: トランザクションデータ + ユーザープロファイル

推奨する技術スタック：
- フロントエンド: [選定理由とともに提案]
- バックエンド: [選定理由とともに提案]
- データベース: [選定理由とともに提案]
- インフラ: [選定理由とともに提案]
```

---

### パターン2: インフラ構築時
```
1. cloud-providers.md でクラウドプロバイダーを選定
2. container-standards.md でコンテナ戦略を確認
3. monitoring-logging.md で監視・ログ基盤を設計
4. インフラ構成図を作成
5. IaCコードを実装
```

**プロンプト例**:
```
以下の標準に従ってKubernetesクラスタを構築してください：
/05-technology-stack/container-standards.md
/05-technology-stack/monitoring-logging.md

要件：
- AWS EKS使用
- Prometheus + Grafanaで監視
- ELK Stackでログ管理
- 開発・ステージング・本番の3環境

実装内容：
1. Terraform でEKSクラスタ構築
2. Helmで監視スタックデプロイ
3. サンプルアプリケーションのデプロイ設定
```

---

### パターン3: データベース設計時
```
1. database-stack.md で推奨データベースを確認
2. データモデルの特性に応じてRDBMS/NoSQLを選択
3. ORMとマイグレーションツールを選定
4. データベーススキーマを設計
5. マイグレーションスクリプトを作成
```

**プロンプト例**:
```

データ要件：
- ユーザー情報（10万レコード）
- トランザクション履歴（100万レコード/年）
- リアルタイムキャッシュ（セッション、ランキング）

推奨構成：
- RDBMS: [選定理由]
- NoSQL: [選定理由]
- ORM: [選定]
- マイグレーションツール: [選定]
```

## 👥 開発チームの利用パターン

2. **監視基盤**: 標準監視スタックの導入

## 📊 技術選定の基準

### 技術評価マトリクス

**評価軸**:
```
1. 成熟度（Production Readiness）
   - 実績、安定性、コミュニティ

2. パフォーマンス
   - 速度、スケーラビリティ

3. 開発効率
   - 学習曲線、ドキュメント、ツールサポート

4. 保守性
   - アップデート頻度、互換性、長期サポート

5. コスト
   - ライセンス費用、運用コスト

6. セキュリティ
   - 脆弱性対応、セキュリティ機能
```

**評価スコア**: 各軸を5段階評価（1〜5点）

---

### 技術承認プロセス

```
1. 提案
   ├─ 新技術の提案書作成
   └─ ユースケースと評価結果を添付

2. 評価
   ├─ パイロットプロジェクト実施
   └─ 技術評価マトリクスで評価

3. レビュー
   ├─ アーキテクトチームレビュー
   └─ セキュリティレビュー

4. 承認
   ├─ CTOまたはテクニカルリード承認
   └─ approved-technologies.mdに追加

5. 導入
   ├─ ドキュメント整備
   ├─ トレーニング実施
   └─ 全チームへアナウンス
```

## 📈 技術ロードマップ

### 短期（0〜6ヶ月）
- 既存プロジェクトの技術スタック棚卸し
- 非推奨技術からの移行計画策定
- 監視・ログ基盤の標準化

### 中期（6〜12ヶ月）
- 新技術の評価と導入（パイロットプロジェクト）
- コンテナ化の全プロジェクト展開
- クラウドネイティブアーキテクチャへの移行

### 長期（12ヶ月〜）
- マイクロサービスアーキテクチャの全面採用
- AIツール（Devin等）との統合強化
- ゼロトラストセキュリティの実装

## 🔄 更新・ガバナンス

### 標準の更新プロセス
1. **四半期レビュー**: 技術スタックの定期的な見直し
2. **新技術評価**: 年2回の新技術評価期間
3. **バージョンアップ**: 主要技術のバージョンアップ計画
4. **非推奨化**: 古い技術の段階的廃止

### バージョンポリシー
- **メジャーバージョン**: LTS（Long Term Support）版を推奨
- **マイナーバージョン**: 安定版リリース後3ヶ月以内に更新検討
- **パッチバージョン**: セキュリティパッチは即座に適用
